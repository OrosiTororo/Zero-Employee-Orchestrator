# CLAUDE.md

## このファイルの役割

このファイルは、Claude Code が CommandWeave リポジトリ内で作業するときの**常時適用ルール**である。  
思想文書や要望一覧をそのまま実装しようとせず、ここに書かれた原則を優先する。

参照優先度は次の通り。

1. この `CLAUDE.md`
2. `IMPLEMENTATION_PLAN.md`
3. `docs/vision/CommandWeave_complete_with_dify_coze_additions.md` などの上位思想文書
4. 既存コードと既存テスト

---

## 1. プロジェクトの一文定義

CommandWeave は、自然言語で業務用 Skill を設計し、複数 AI を組織的に協働させ、人間の承認と監査可能性を前提に業務実行を支援する AI オーケストレーション基盤である。

---

## 2. 最上位の設計原則

### 2.1 AI を単体ツールではなく「組織」として扱う
- 単一 AI の一発実行より、計画・実行・検証・再提案の流れを優先する。
- 実装時は、将来的な Agent Teams 的な拡張を妨げない構造を優先する。

### 2.2 人間の最終承認を外さない
以下は**必ず承認必須**として扱う。
- 公開
- 投稿
- 課金
- 削除
- 権限変更
- 外部送信

上記に該当しうる処理は、勝手に自動実行する実装にしない。

### 2.3 ブラックボックスを減らす
最低限、ユーザーが次を追跡できる設計を優先する。
- どの工程が走ったか
- どの AI / Skill / Plugin が担当したか
- なぜ停止・差し戻し・再計画が起きたか
- どのモデル / API / 外部接続を使ったか

### 2.4 最新技術は本体に抱え込みすぎない
- 短寿命なモデル固有実装や業務固有ロジックは、可能な限り Plugin / Extension 側に逃がせる構造にする。
- 本体は、計画・承認・実行・監査・権限・履歴の中核に集中する。

### 2.5 安全性と信頼性を最優先する
- 誤作動時は安全側に倒す。
- 失敗時は「無理に続行」より「安全停止」を選ぶ。
- 危険操作は必ず事前明示する。

---

## 3. 用語の固定定義

### Skill
AI が単一タスクを実行するための最小単位の能力。  
例: 翻訳、分析、台本生成、ファイル整理。

### Plugin
複数の Skills や補助ツールを束ね、特定業務を成立させる機能パッケージ。  
例: YouTube Plugin、ブログ運用 Plugin。

### Extension
本体の UI、接続先、設定、通知、開発体験など、ソフト全体の利便性を広げる拡張。  
例: OAuth 対応、MCP 接続、通知機能。

### spec / plan / tasks
CommandWeave の中間成果物。
- `spec`: 何を達成するか、目的、制約、受け入れ条件
- `plan`: どう実現するか、工程、利用 AI、承認ポイント
- `tasks`: 実行可能な作業単位、依存関係、完了条件

この定義を勝手に崩さないこと。

---

## 4. 本体と拡張の境界

### 本体に置くもの
- 自然言語入力の受付
- Design Interview の起動と管理
- spec / plan / tasks の保存
- 承認フロー
- 実行状態管理
- 実行ログ / 監査ログ
- 権限モデル
- 危険操作のゲート
- Plugin / Skill / Extension のメタ情報管理

### Plugin / Extension 側に逃がすもの
- 業界固有ワークフロー
- 特定 SaaS 連携の詳細ロジック
- モデル固有最適化
- コーディング特化や SNS 特化などの領域別機能
- 頻繁に変化する業務ロジック

新規機能を実装する際は、**まず本体に入れるべきかを疑うこと**。

---

## 5. 実装時の必須姿勢

### 5.1 小さく刻む
- 一度に大きく壊さない。
- 小さな差分で進める。
- まず動く最小実装を作る。

### 5.2 既存構造を尊重する
- 既存ファイルや既存責務を無視して新構造を乱立させない。
- 大規模移動や全面書き換えは、必要性が高い場合に限る。

### 5.3 変更理由を説明可能にする
- なぜその設計にしたかを README / DESIGN / コメント / PR 文脈で追えるようにする。
- 監査可能性を下げる抽象化は避ける。

### 5.4 失敗に備える
- 失敗時の状態を曖昧にしない。
- `pending / approved / running / blocked / failed / completed` などの状態遷移を明示する。
- エラー時は「何が起きたか」が追跡できる構造化ログを優先する。

## 6. UX / UI 方針

- 非エンジニアでも、計画・承認・実行・差し戻しの流れが分かる UI を優先する。
- CLI / TUI / GUI のどれであっても、見せる情報の本質は共通にする。
- 少なくとも以下を見える化する。
  - 現在の工程
  - 担当 AI / Skill
  - 必要権限
  - コスト概算
  - 危険操作の有無
  - 承認待ちポイント
  - 差し戻し理由

---

## 7. セキュリティとデータ取り扱い

### 絶対に守ること
- API キー、OAuth トークン、シークレットは平文保存しない。
- ログに秘密情報を直接出さない。
- 外部共有時は個人情報・機密情報・認証情報を除外する。
- 危険な Plugin / Skill / Extension を無条件で有効化しない。

### 優先すること
- 最小権限
- 分離保存
- サニタイズ
- 監査可能性
- ローテーションしやすさ

---

## 8. Claude Code がやってはいけないこと

- 思想文書の全項目を一気に実装しようとすること
- 承認必須操作を自動化前提で実装すること
- 本体に業務固有ロジックを大量に埋め込むこと
- ログや履歴を雑にして監査可能性を落とすこと
- 既存の責務分離を壊してまで見た目だけ整えること
- 将来拡張のためと言って、今不要な抽象化を入れすぎること
- モックだけ整えて実フローが繋がらない状態で完了扱いにすること

---

## 9. 実装の優先順位

常に次の順で考えること。

1. MVP の end-to-end を通す
2. 権限表示と承認フローを整える
3. 実行ログ・監査ログを整える
4. Skill / Plugin 境界を整える
5. 失敗時の停止・再試行・再提案を整える
6. Registry や高度共有機能は後回しにする

## 10. 望ましい出力スタイル

Claude Code は、作業時に次を明確に示すこと。

- 何を変更したか
- なぜ変更したか
- どの原則に基づくか
- どこまで完了したか
- 未実装部分は何か
- 次の最小ステップは何か

可能なら、変更後に次を簡潔にまとめる。
- affected files
- acceptance criteria の達成状況
- 残課題

---

## 11. 最初の完成目標

最初の完成目標は、CommandWeave の本質である次の流れを最小構成で通すこと。

1. 自然言語で依頼を受ける
2. Design Interview で要件を深掘る
3. spec / plan / tasks を保存する
4. 計画に権限・コスト・危険操作を表示する
5. ユーザー承認を受ける
6. 実行する
7. Judge で最低限の品質確認を行う
8. ログ・成果物・履歴を保存する

この流れを壊す変更は避けること。

---

## 12. 実装判断で迷ったときの基準

迷った場合は次を順番に自問すること。

1. それは本体機能か、Plugin / Extension か
2. 安全性、権限、監査を弱めていないか
3. 非エンジニアにも扱えるか
4. ブラックボックス化を進めていないか
5. 今の MVP に必要か
6. 将来拡張のためと言って過剰設計していないか

上記に自信を持って答えられない場合は、**より小さく、安全で、説明可能な案**を採用すること。
