## 1. 用語の定義

### 1.1 Skill / Plugin / Extension の違い

CommandWeave では、機能追加の単位を次の3層で整理する。

#### Skill
AI が特定タスクを実行するための最小単位の能力。  
プロンプト、スクリプト、手順、制約、利用ツールなどを含む。

- 例: ファイル整理、台本生成、翻訳、競合分析
- 役割: 単一タスクの実行能力

#### Plugin
複数の Skills や補助ツールをまとめて配布・管理するパッケージ。

- 例: YouTube Plugin、ブログ運用 Plugin、AI秘書 Plugin
- 役割: 特定業務を実現するための機能セット

#### Extension
アプリ本体の動作環境や UI、接続先、開発体験を拡張する仕組み。

- 例: VS Code 風 UI、MCP 接続、OAuth 対応、通知機能、クラウド連携
- 役割: ソフト全体の利便性・開発体験・接続性の向上

### 1.2 まとめ表

| 項目 | Skill | Plugin | Extension |
|---|---|---|---|
| 主な対象 | AI の実行能力 | 業務機能のまとまり | アプリ全体の拡張 |
| 単位 | 最小 | 中間 | 広範 |
| 内容 | 単一タスクの手順・知識 | Skills とツールの詰め合わせ | UI、接続、開発体験、設定の強化 |
| 例 | 翻訳、分析、整理 | YouTube Plugin | VS Code風 UI、MCP、通知 |

### 1.3 重要な前提

- Skills を単体で追加するだけでなく、既存 Skills や新規 Skills を組み合わせて Plugin として追加できるようにする。
- Plugin は VS Code の拡張機能に近い感覚でインストール・削除・共有・公開できるようにする。
- CommandWeave は、チャットによる自然言語だけで Skill / Plugin を設計・生成・修正できることを目標とする。
- 共有・公開時には個人情報や機密情報を含めない設計が必要である。

## 2. プロジェクト要約

### 2.1 結論

CommandWeave は、自然言語による Skill 設計とマルチ AI 協調実行に対応した、軽量かつ拡張性の高い汎用業務オーケストレーションシステムである。  
エンジニア向けには CLI / TUI、非エンジニア向けには GUI を提供し、どちらも共通の AI 組織基盤の上で動作する。

### 2.2 目的

CommandWeave の目的は、マルチ AI エージェントと人間が共存・協調しながら、計画、実行、検証、改善を一つの環境で回せる AI 組織基盤を実現することである。

特に重視するのは次の3点である。

- AI を単体ツールではなく「組織」として運用できること
- AI の判断や行動をブラックボックス化させず可視化すること
- 人間の最終承認を残しつつ、業務の大部分を自律実行できること

### 2.3 代表ユースケース

最も先鋭的なユースケースは、経営者 1 名が指示と承認のみを行い、実務は AI 組織が担う「ゼロパーソンカンパニー」の運営である。  
ただし CommandWeave はその専用ソフトではなく、以下のような幅広い用途に対応する汎用基盤として設計する。

- 会社業務全般の自律化
- AI 研究・実験
- コーディング支援
- マーケティング・SNS 運用
- クリエイティブ制作
- バックオフィス補助
- 一般ユーザー向け AI 自動化
- 将来的な AI ロボット・スマート機器・遠隔操作環境

## 3. CommandWeave の中核思想

### 3.1 AI を「道具」ではなく「組織」として扱う

CommandWeave は、単一の AI チャットや単発自動化ではなく、複数の AI が役割分担し、相互監視し、情報共有しながら動く AI 組織を構築することを目指す。

### 3.2 人間の最終承認を外さない

重要操作、外部送信、公開、課金、権限付与、プライバシーに関わる処理については、必ず人間の確認と承認を必要とする。

### 3.3 ブラックボックスを減らす

- どの AI が何を担当しているか
- 何を根拠に計画したか
- どこまで完了したか
- どのモデルや API を使ったか
- なぜ差し戻されたか

これらをユーザーが確認できるようにする。

### 3.4 最新技術は本体ではなく拡張で取り込む

アプリ本体の更新頻度は抑えつつ、最新技術や新しい業務対応は Skills / Plugins / Extensions 側で追加できる構造を重視する。

### 3.5 安全性と信頼性を最優先する

CommandWeave は、AI を無制限に自律化するためのソフトではなく、権限、監査、ルール、承認を備えた「高信頼な AI 組織運営基盤」を目指す。

## 4. 利用時の基本フロー

1. ユーザーが自然言語で目的や業務を入力する。  
2. AI 組織が要件を深掘りし、実行計画を作成する。  
3. 計画には工程、担当 AI、想定時間、推定 API コスト、必要権限、外部接続の有無を含める。  
4. ユーザーが計画を確認し、修正・追加・削除を指示する。  
5. 承認後、AI 組織が業務を実行する。  
6. 実行中は進捗、成果物、失敗、自己修復履歴を可視化する。  
7. 完了後、成果物をユーザーが確認し、差し戻しまたは承認する。  
8. 会話・判断・成果物・改善履歴を記録し、次回以降の運用に活かす。

## 5. 現状の実装イメージ

### 5.1 全体像

CommandWeave は、AI の進歩に柔軟に対応できる次世代の AI 統合環境であり、CLI / TUI と GUI の両方から同じ AI 組織を操作できる構造を目指す。

### 5.2 想定する特徴

- ローカル常駐デスクトップでも仮想サーバー上でも動作可能
- 既存の AI ツール、API、ワークフローに幅広く接続可能
- エンジニアだけでなく非エンジニアも利用できる
- 長期運用を前提に、軽量・拡張性・移植性を重視する
- 単発自動化ではなく、業務全体の連続運用を支援する

### 5.3 代表的な検証テーマ

YouTube 運営を題材に、以下のような複雑な業務を AI 組織で処理できるかを検証する。

- トレンド分析
- 台本作成
- 競合調査
- パフォーマンス分析
- 次の一手の提案

この YouTube 領域は、製品の本質そのものではなく、複雑な業務ワークフローを扱えるかを確かめるための代表的な Plugin 領域と位置づける。

## 10. 要望の整理

以下は元文書の要望を、重複を減らしながら論点別に整理したものである。

### 10.1 理念・方向性

- システム本体の更新頻度は抑え、最新技術は Skills / Plugins 側で吸収したい。
- AI が AI を指揮するオーケストレーションを実現したい。
- 単機能な AI SaaS ではなく、企業全体の業務をシームレスに扱える基盤にしたい。
- セキュリティ、権限、保守運用まで含めて企業利用に耐える必要がある。
- 企業や個人が、自分たちの現場に特化した AI を容易に作れる環境にしたい。

### 10.2 最重要目的

CommandWeave の最重要目的は、既存 AI エージェントの最大の課題である「リスク管理と信頼構造の欠如」を解消することである。

具体的には、以下に対応できる基盤を目指す。

- 誰がどの AI に何を任せたかを追跡できること
- 過剰権限を避けること
- 操作ログを明確に残すこと
- 退職・担当変更・運用停止時にも制御できること
- ERP / CRM / セキュリティのような慎重さが必要な領域とも接続できること

### 10.3 参照したい外部思想

CommandWeave は、OpenCode の以下の思想を継承対象とする。

- 軽量・高速
- CLI / TUI ベースでも使いやすい
- 低スペック環境でも動きやすい
- 柔軟なモデル選択
- MCP 対応
- 多数の LLM Provider への接続
- LSP 統合
- セッション共有
- スクリプトとの統合

その上で、OpenCode と異なり、非エンジニアでも扱える GUI を追加し、VS Code のような UI/UX に近づける。

### 10.4 UI / 使用感の要望

- GUI は VS Code 風、CLI / TUI は OpenCode 風にする。
- Skills / Plugins を一覧から追加・削除・確認できるようにする。
- チャットから業務用の Skills / Plugins を作成・修正できるようにする。
- 業務状況、進捗、コスト、使用 API をユーザーが常に確認できるようにする。
- AI が API を自動選定しても、ユーザーがコスト重視・品質重視・無料枠重視を選べるようにする。
- ローカル・クラウド・各種サービスに広く接続できるようにする。

### 10.5 セキュリティ・懸念点

- 最低限どの機能を本体に入れ、何を Plugin 化するかの基準を明確にする必要がある。
- Skill / Plugin をチャットで生成できても、安全性検証なしには導入させない。
- 大規模システムの置き換えでは、要件充足と保守運用の証明が必要になる。
- 同意画面フィッシングや危険な設定ファイルを含むリポジトリの取り扱いに注意する。
- Skill マージ前に静的解析や秘密情報漏洩チェックを入れる。
- OAuth は初期設定で必須にせず、必要に応じて Extension で有効化する方向も検討する。

### 10.6 AI エージェント運用の要望

- Agent Teams 型のマルチ AI 組織を基本とする。
- 役割ごとに班や部署を分け、双方向通信や共有タスクリストで連携できるようにする。
- AI 秘書、SNS 運用、リモート操作、ブログ・動画・音声の横展開などの高頻度業務を Plugin 化したい。
- ユーザーの価値観や判断傾向を学習して、分身 AI として育つ方向を目指したい。
- 常時稼働・長期記憶・提案型の AI 組織を実現したい。

### 10.7 ログイン・接続の要望

- OpenRouter 固定ではなく、専用アカウントや Google アカウントなど柔軟なログイン方式を持たせたい。
- OpenRouter 以外の AI API アグリゲーターやローカル LLM にも対応したい。
- Claude Code、Codex、GitHub Copilot などの既存サービスを有効活用したい。
- ローカル LLM や無料枠を活用し、API コストゼロ運用も選べるようにしたい。

### 10.8 将来方針

- 無料・軽量・高機能な AI エージェント版 VS Code のような位置づけを目指す。
- コーディングに限らず、知識作業や現場業務にも対応する。
- 将来的には AI ロボット、スマートグラス、AR / VR、ハードウェア開発にも転用したい。
- 法律、税務、経済など専門ナレッジを扱う部署型 AI も視野に入れる。

### 10.9 メタスキルの導入

現在の AI エージェントには、ハードスキルやソフトスキルはあっても、学び方・見通し方・自己修正のようなメタスキルが弱い。  
CommandWeave では、AI 組織に以下の能力を持たせたい。

- 学習能力
- 批判的思考
- 自己認識
- 好奇心
- システム思考
- 実行と改善の往復能力

### 10.10 並列開発への示唆

Claude Code の並列開発事例から、次の方針を参考にする。

- 実装前の計画に十分な時間をかける
- AI の行動ルールを `CLAUDE.md` などで明文化する
- `git worktree` などを使って並列実装を衝突なく進める
- テストや検証は Skill 化して再利用可能にする

### 10.11 Plugin 戦略

CommandWeave は、最初からすべてを詰め込むのではなく、最低限の中核機能の上に必要な Plugin を追加して広げる構造を取る。

初期状態では以下が重要になる。

- 汎用 Skill 作成能力
- Plugin の追加・削除・共有・公開
- Skill / Plugin の安全性検査
- 公式 Plugin のいくつかの同梱
- ユーザーごとの最適化余地の確保

想定 Plugin 例:

- YouTube / SNS 分析・投稿
- AI 秘書
- iPaaS 連携
- ローカル AI 対応
- ブログ・動画・音声の変換
- OpenClaw 型の常時稼働エージェント連携
- コーダーエディタ追加

6. 長期運用とコスト最適化に耐えるか。  
7. 人間の最終承認を残しているか。
