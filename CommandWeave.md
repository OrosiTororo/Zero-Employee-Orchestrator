## 1. 用語の定義

### 1.1 Skill / Plugin / Extension の違い

CommandWeave では、機能追加の単位を次の3層で整理する。

#### Skill
AI が特定タスクを実行するための最小単位の能力。  
プロンプト、スクリプト、手順、制約、利用ツールなどを含む。

- 例: ファイル整理、翻訳、台本生成、競合分析
- 役割: 単一タスクの実行能力

#### Plugin
複数の Skills や補助ツールをまとめて配布・管理するパッケージ。  
特定業務を成立させるための機能セットであり、VS Code の拡張機能に近い感覚で扱う。

- 例: YouTube Plugin、ブログ運用 Plugin、AI 秘書 Plugin
- 役割: 特定業務を実現するためのまとまり

#### Extension
アプリ本体の動作環境、UI、接続先、開発体験を拡張する仕組み。

- 例: VS Code 風 UI、MCP 接続、OAuth 対応、通知機能、クラウド連携
- 役割: ソフト全体の利便性・接続性・開発体験の向上

### 1.2 まとめ表

| 項目 | Skill | Plugin | Extension |
|---|---|---|---|
| 主な対象 | AI の実行能力 | 業務機能のまとまり | アプリ全体の拡張 |
| 単位 | 最小 | 中間 | 広範 |
| 内容 | 単一タスクの手順・知識 | Skills と補助ツールの詰め合わせ | UI、接続、設定、開発体験の強化 |
| 例 | 翻訳、分析、整理 | YouTube Plugin | VS Code 風 UI、MCP、通知 |

### 1.3 重要な前提

- Skills を単体で追加するだけでなく、既存 Skills や新規 Skills を組み合わせて Plugin として追加できるようにする。
- Plugin は VS Code の拡張機能のように、インストール・削除・共有・公開できるようにする。
- CommandWeave は、チャットによる自然言語だけで Skill / Plugin を設計・生成・修正できることを目標とする。
- 共有・公開時には個人情報、機密情報、認証情報を含めない設計が必要である。
- ローカル保存、Git 管理、クラウド保存は用途に応じて選択可能にする。
- AI による Git 管理、ローカル保存、GitHub や Google Cloud などのクラウド保存を運用方針に応じて切り替えられるようにする。
- 共有・公開される Skills / Plugins は最新版取得や更新確認ができるようにする。

## 2. プロジェクト要約

### 2.1 一文定義

CommandWeave は、自然言語で業務用 Skill を設計し、複数 AI を組織的に協働させ、人間の承認と監査可能性を前提に業務実行を支援する AI オーケストレーション基盤である。

### 2.2 結論

CommandWeave は、自然言語による Skill 設計とマルチ AI 協調実行に対応した、軽量かつ拡張性の高い汎用業務オーケストレーションシステムである。  
エンジニア向けには CLI / TUI、非エンジニア向けには GUI を提供し、どちらも共通の AI 組織基盤の上で動作する。

### 2.3 目的

CommandWeave の目的は、マルチ AI エージェントと人間が共存・協調しながら、**計画、実行、検証、改善、監査、再利用**を一つの環境で回せる AI 組織基盤を実現することである。

特に重視するのは次の5点である。

- AI を単体ツールではなく「組織」として運用できること
- AI の判断や行動をブラックボックス化させず可視化すること
- 人間の最終承認を残しつつ、業務の大部分を自律実行できること
- 権限、監査、保守運用を含めて企業利用に耐えること
- 新しい AI サービス、API、モデル変化に長期的に追従できること
- 既存 AI エージェントが抱える過剰権限、監査不透明、継続稼働リスクに企業が対応できること

### 2.4 代表ユースケース

最も先鋭的なユースケースは、経営者 1 名が指示と承認のみを行い、実務は AI 組織が担う**ゼロパーソンカンパニー**の運営である。  
ただし CommandWeave はその専用ソフトではなく、以下のような幅広い用途に対応する汎用基盤として設計する。

- 会社業務全般の自律化
- AI 研究・実験
- コーディング支援
- マーケティング・SNS 運用
- クリエイティブ制作
- バックオフィス補助
- 一般ユーザー向け AI 自動化
- 将来的な AI ロボット、スマート機器、遠隔操作環境

### 2.5 ゼロパーソンカンパニーとの関係

ここで言うゼロパーソンカンパニーとは、**社長という意志決定者は存在するが、組織として多数の人員を雇わずに会社を回せる状態**を指す。  
CommandWeave は、その状態を成立させるために必要な**計画・実行・監査・承認・改善**の仕組みを、一つのソフトウェア環境に統合することを目指す。

### 2.6 CommandWeave が対象とする利用者層

CommandWeave は、単なる開発者向けツールでも、単なる一般ユーザー向け自動化ツールでもない。  
主な対象は次の3層である。

- **非エンジニアの実務者・経営者**  
  自然言語で業務を依頼し、AI 組織の計画・実行・承認を管理したい利用者
- **エンジニア・研究開発者**  
  Skills / Plugins / Extensions を開発し、業務自動化基盤を拡張したい利用者
- **将来的な組織利用者**  
  監査、権限管理、承認、履歴保存を前提に AI を業務に導入したい組織

そのため CommandWeave は、**GUI だけでも使えること**と、**CLI / TUI でも深く制御できること**の両立を目指す。

## 3. CommandWeave の中核思想

### 3.1 AI を「道具」ではなく「組織」として扱う

CommandWeave は、単一の AI チャットや単発自動化ではなく、複数の AI が役割分担し、相互監視し、情報共有しながら動く AI 組織を構築することを目指す。

### 3.2 人間の最終承認を外さない

重要操作、外部送信、公開、投稿、課金、権限変更、プライバシーに関わる処理については、必ず人間の確認と承認を必要とする。

### 3.3 ブラックボックスを減らす

以下を常にユーザーが確認できるようにする。

- どの AI が何を担当しているか
- 何を根拠に計画したか
- どこまで完了したか
- どのモデルや API を使ったか
- なぜ差し戻されたか
- なぜ再計画が発生したか
- どの Skill / Plugin / Extension が使われたか

### 3.4 最新技術は本体ではなく拡張で取り込む

アプリ本体の更新頻度は抑えつつ、最新技術や新しい業務対応は Skills / Plugins / Extensions 側で追加できる構造を重視する。

### 3.5 安全性と信頼性を最優先する

CommandWeave は、AI を無制限に自律化するためのソフトではなく、権限、監査、ルール、承認を備えた**高信頼な AI 組織運営基盤**を目指す。

### 3.6 OS と呼ぶ理由

CommandWeave は、単体タスクの自動化だけでなく、業務同士の接続、成果物の再利用、履歴管理、権限管理、再計画、監査、拡張導入までを一つのアプリケーション上で扱う。  
そのため、単発の AI エージェントではなく、**業務全体を支える OS 的基盤**として位置づける。

### 3.7 Agent Teams と Subagents の考え方

CommandWeave では、複数 AI の協働を単純な「サブエージェント呼び出し」で終わらせず、**役割分担されたチーム運用**として扱うことを重視する。

#### Subagents
- 主エージェントから指示を受けて部分作業を行う
- 一方向の報告関係になりやすい
- 単純な補助、委譲、短い処理に向く

#### Agent Teams
- 複数 AI が役割ごとに担当を持つ
- 相互に情報共有、確認、批評、再提案を行う
- 複雑な業務、長い工程、複数成果物、継続運用に向く

CommandWeave は、単なる subagent 呼び出しではなく、**計画・実行・監督・検証を分担した Agent Teams 的な構造**を重視する。  
また将来的には、ユーザーが**チーム単位だけでなく個別メンバー単位でも介入・差し戻し・指示変更できる**ことを目指す。

## 4. 利用時の基本フロー

1. ユーザーが自然言語で目的や業務を入力する。  
2. AI 組織が要件を深掘りし、実行計画を作成する。  
3. 計画には工程、担当 AI、想定時間、推定 API コスト、必要権限、外部接続の有無、危険操作の有無を含める。  
4. ユーザーが計画を確認し、修正・追加・削除を指示する。  
5. 権限チェック、危険操作警告、コスト確認を経たうえで承認された場合のみ実行に進む。  
6. 実行中は進捗、成果物、失敗、自己修復履歴、利用中の Skill / Plugin / Extension を可視化する。  
7. 完了後、成果物をユーザーが確認し、差し戻しまたは承認する。  
8. 会話、判断、成果物、改善履歴を記録し、次回以降の運用に活かす。  

### 4.0.1 spec / plan / tasks 形式での中間成果物管理

CommandWeave では、実行前の計画を単なる会話ログとして終わらせず、必要に応じて中間成果物として構造化保存する。  
参考にできる実践として、次の3段階を採用する。

- **spec**: 何を達成するか、目的、制約、受け入れ条件
- **plan**: どう実現するか、工程、利用する AI / Skill / Plugin / 外部接続、承認ポイント
- **tasks**: 実行可能な作業単位、担当、依存関係、検証条件、完了条件

ただし CommandWeave では、これを**ソフトウェア開発専用の設計書形式ではなく、汎用業務にも使える計画記述形式**として扱う。  
たとえば、SNS 運用、競合調査、資料作成、バックオフィス処理でも、目的・制約・工程・担当・検証条件・人間承認ポイントを `spec / plan / tasks` に準じた形で記録できるようにする。

これにより、会話依存のブラックボックス性を減らし、再利用性、監査性、差し戻しや再実行のしやすさを高める。

### 4.1 計画時に明示すべき追加情報

CommandWeave の計画提示では、工程、担当 AI、費用、権限、危険操作だけでなく、必要に応じて次の情報も明示する。

- 使用予定モデルの候補と選定理由
- 高品質重視 / 速度重視 / 低コスト重視 / 無料枠優先などの実行モード
- 失敗時の代替ルート
- 外部参照元の種類
- ローカルのみで完結する工程と外部接続が必要な工程の区別
- 実行後に人間承認が必要な箇所の一覧

これにより、ユーザーは「何が起きるか分からないまま実行する」状態を避けられる。

---

### 4.2 自然言語トリガーと内部コマンドモデル

CommandWeave では、自然言語だけで基本操作を進められることを重視する。  
その一方で、繰り返し使う操作については、自然言語と内部コマンド体系を対応づけることで一貫性を高める。

たとえば内部的には、次のような操作概念を持てる。

- 依頼内容から Interview を始める
- 計画化を始める
- 承認待ちに移す
- 実行を開始する
- 次のタスクだけ進める
- レビューを要求する
- 完了処理に進める
- 履歴や進捗を同期する

ただし CommandWeave では、これを**CLI 専用の開発者操作体系に閉じず、GUI と自然言語の両方から同じ意味で扱える操作モデル**として実装する。

## 5. 使用時の想定挙動

- ユーザーは AI と対話しながら、業務の目的、条件、制約、優先順位を詰める。
- AI 組織は内部情報と外部情報を参照しながら、実行計画を立案する。
- 計画時には、どの工程でどの AI や API を使うか、どの程度の費用がかかるか、どの権限が必要かを明示する。
- ユーザーはその計画に対して批評や修正を加え、納得した場合のみ実行を許可する。
- 実行時には役割や工程ごとに AI が担当班として割り振られ、進行状況や課題を相互に監視する。
- その過程をブラックボックス化せず、思考の要約、判断根拠、実行ログ、差し戻し理由をユーザーが確認できるようにする。
- 実行中の追加指示や修正依頼にも対応できるようにする。
- 納品、投稿、公開、外部送信などの最終操作は必ず人間が承認する。
- 実行履歴、会話履歴、判断履歴、成果物は記録し、将来的な分身 AI や AI 秘書、経験学習に活用できるようにする。
- 計画時には必要に応じて Skills を組み合わせたり新規作成したりして、タスク専用の Plugin を構築できるようにする。
- Skills や Plugins は、個人情報や機密情報を除いたうえで共有・公開可能にする。
- コンテキストウィンドウの制限に対応するため、履歴管理機構と必要情報の再取得機構を設ける。

## 6. MVP 定義

### 6.1 MVP の目的

MVP の目的は、CommandWeave の本質である**自然言語による業務設計、複数 AI による計画・実行・検証、人間承認、可視化された運用**を、最小構成で end-to-end に成立させることである。

### 6.2 MVP に必須の機能

- 自然言語入力による業務依頼
- Design Interview による要件深掘り
- Plan / DAG の生成
- API コストの概算提示
- 必要権限と外部接続の提示
- Skill 割当と基本的な Skill 実行
- Judge による品質確認
- 差し戻しと再提案
- 人間承認フロー
- 実行ログ・成果物・履歴の保存
- GUI での確認画面
- 最低限の認証情報保護
- Local Context の安全な読み込み
- 実証用 Plugin を 1 系統以上搭載
- 本体機能と Plugin / Extension の境界が最低限整理されていること

### 6.3 MVP で対象外とするもの

- 大規模な公開 Registry 経済圏の完成
- 完全自律での無承認外部実行
- 大量の企業向け権限階層の完全実装
- すべての業務領域への対応
- 高度な分身 AI の完成
- 大規模なマルチテナント運用
- 高度なロボット制御や物理機器連携
- 外部 SaaS の認証情報自動再発行

### 6.4 MVP の成功条件

- ユーザーが自然言語だけで業務依頼を開始できる
- AI 組織が実行計画を提示し、ユーザーが修正・承認できる
- 実行から成果物生成まで一連の流れが通る
- Judge によって最低限の品質確認が行われる
- 外部送信や重要操作が人間承認なしに進まない
- 進捗と実行内容がユーザーに見える
- 失敗時に再提案または安全停止できる

### 6.5 MVP における代表デモ条件

MVP は、概念説明ではなく実動デモとして成立している必要がある。  
少なくとも次の条件を満たすことを目標とする。

- 自然言語入力から Design Interview が開始する
- AI 組織が計画と担当分担を提示する
- コスト、権限、危険操作の有無が表示される
- 実行中に進捗と成果物が見える
- Judge による品質判定が行われる
- 差し戻し時に再提案または再実行が起こる
- 最後の公開、送信、投稿は人間承認を要求する
- 実行後に履歴、成果物、判断情報が保存される

## 7. 権限モデル

CommandWeave では最低限、以下のロールを想定する。

### 7.1 Owner
- 全権限を持つ
- システム設定、認証情報管理、Plugin 導入、公開設定、承認、監査ログ閲覧が可能

### 7.2 Admin
- 組織設定、Plugin 導入、一部承認、監査ログ閲覧が可能
- 認証情報の一部編集権限を持つ場合がある

### 7.3 User
- 業務依頼、計画確認、成果物確認、差し戻しが可能
- 危険権限の付与や外部公開はできない

### 7.4 Auditor
- 実行履歴、承認履歴、監査ログの閲覧が可能
- 実行や編集権限は持たない

### 7.5 Developer
- Skill / Plugin / Extension の作成や修正が可能
- 本番導入や公開には別途承認を必要とする

## 8. データ保存・メモリ方針

### 8.1 保存対象

- 会話履歴
- 実行計画
- 実行ログ
- 承認履歴
- 成果物
- エラー履歴
- 利用した Skill / Plugin / Extension の情報
- ユーザー設定
- 失敗分類と回復履歴

### 8.2 保存対象外または厳格管理対象

- API キー
- OAuth トークン
- 決済情報
- 個人情報
- 社外秘情報
- 外部共有してはいけない内部文書

これらは暗号化・分離保存し、通常ログには平文で残さない。

### 8.3 メモリ利用方針

- Experience Memory は、成功した計画や差し戻しから学んだ改善パターンを保存する。
- 分身 AI や AI 秘書に転用する記録は、ユーザーが保存範囲を制御できる設計とする。
- メモリには削除、リセット、エクスポート機能を設ける。
- 共有・公開時には個人情報、機密情報、認証情報を除外する。

### 8.4 保存・共有時の情報サニタイズ方針

CommandWeave では、保存・共有・公開の各段階で、情報をそのまま扱わない。  
少なくとも次の処理方針を取る。

- API キー、OAuth トークン、シークレット値はマスキングまたは参照 ID 化する
- 個人名、住所、決済情報などの個人情報は共有時に除去または匿名化する
- 社外秘資料、未公開情報、契約情報は公開対象から除外する
- 会話ログや改善ログを学習や共有に使う場合は、ユーザーが範囲を制御できるようにする

共有文化を成立させる前提として、**安全なサニタイズ処理**は中核要件である。

---

### 8.5 改善知識の分離保存方針

CommandWeave では、単なる会話履歴の保存だけでなく、**改善知識を用途ごとに分離して保存する**ことを重視する。  
たとえば次のように整理できる。

- **Experience Memory**: 成功した実行パターン、再利用可能な判断
- **Failure Taxonomy**: 失敗分類、よくある差し戻し理由、再発防止策
- **Policy Memory**: 承認条件、禁止事項、権限ルール
- **Skill Improvement Memory**: 特定 Skill の改善点、実務上の癖、品質基準
- **Plugin Operation Memory**: 業務単位での運用ノウハウ、必要な確認順序

この設計により、単に「前回の会話を思い出す」だけではなく、**改善がどの層に属するかを明示したうえで再利用**できるようにする。

## 9. 実行環境と責任分界

### 9.1 ローカル側で担うもの

- GUI / CLI / TUI
- 設定管理
- ローカル状態管理
- 認証情報の保持
- ローカルファイル参照
- 監査ログ表示
- ユーザー承認 UI

### 9.2 クラウド側または外部サービスが担うもの

- LLM 推論
- 外部 API 連携
- クラウド保存
- モデル更新情報取得
- 外部ソース検索
- 将来的な共有 Registry の配布基盤

### 9.3 運用方針

- 本体は長期安定運用を優先し、更新頻度は必要最小限に抑える。
- 新しい業務対応や最新技術への追従は、原則として Skill / Plugin / Extension 側で吸収する。
- 利用規約・接続方式・提供形態が変化しうる外部 AI サービスについては、公式情報を最優先で参照し、非公式かつ危険な接続を避ける。
- 重要な料金判断、プライバシー判断、外部送信判断は、必ず計画段階で人間確認を経る。

- ローカル保存とクラウド保存を選択可能にする。
- オフライン時は実行可能な機能を明示し、外部接続が必要な機能は使用不可として安全に停止する。
- GitHub や Google Cloud などへの保存は明示的な設定と承認のもとで行う。
- AI エージェントによる Git 管理は可能とするが、push、release、公開操作には人間承認を必須とする。
- 社内 LAN、VPN、プロキシ環境下での接続も想定し、証明書や通信経路の扱いを Extension で補えるようにする。

### 9.4 配布と更新の基本方針

CommandWeave は、最終的に一般ユーザーや非エンジニアにも扱える配布形態を目指す。  
そのため、開発環境で動くだけでなく、以下を重視する。

- インストーラーによる導入
- バージョン管理された配布物
- 更新内容の明示
- 将来的な自動アップデート機構
- 更新前のバックアップやロールバック可能性
- 本体更新と Plugin / Extension 更新の切り分け

特に本体は安定性を重視し、業務対応や新機能追加は Plugin / Extension 側で吸収できる構造を優先する。

### 9.5 可観測性の責任分界

CommandWeave では、ローカル / クラウド / 外部サービスの責任分界に加え、**可観測性の責任分界**も明確にする。

- ローカルで記録すべき実行イベント

これにより、進捗把握、障害分析、監査、改善サイクルを同じ運用基盤の中で扱いやすくする。

## 11. 実装・運用で追加考慮すべき技術要件

以下は設計上の重要論点である。

### 11.1 モデル接続と切替

- OpenRouter に閉じず、複数 Provider を直接または AI Gateway / Proxy 経由で接続できるようにする。
- Bedrock、Azure OpenAI、Vertex AI、Cerebras、Cloudflare AI、Z.ai、Ollama、LM Studio などへの接続可能性を考慮する。
- Claude Code や Codex のような公式 CLI / 公式接続は、利用規約に沿う範囲で統合利用できるようにする。
- ローカルモデル、API モデル、サブスク系公式ツールを併用し、コスト・速度・品質・規約適合性のバランスを取る。

- OpenRouter だけに固定せず、複数の LLM Provider、AI Gateway、ローカル LLM に対応可能な構造を維持する。
- LiteLLM Gateway などを用いて、OpenAI 互換形式で複数モデルを切替可能にする。
- 高品質重視、速度重視、無料枠重視、低コスト重視などのモード選択を可能にする。
- モデル変更時にアプリ側の大規模改修を必要としない設計を採る。

### 11.2 AI Gateway / コスト管理

- API コスト見積りだけでなく、キャッシュ、利用制限、失敗時の代替提案も考慮する。
- モデル料金、特性、更新情報を把握するための共通データ置き場を持つ。
- 情報更新そのものが API コストを増やしすぎないようにする。
- AI 自体が無駄な API 呼び出しを減らすよう改善できる構造を持つ。

### 11.3 プロキシ・企業ネットワーク対応

- 社内プロキシや VPN を経由しないと接続できない環境を想定する。
- 証明書エラーや接続制限に対応するための設定層を用意する。
- これらは本体に過剰に埋め込むのではなく、Extension で補強可能にする。

### 11.4 既存 AI サービスの活用

- 既存の AI サービスやサブスクリプションを無駄なく活用できることを重視する。
- 非公式なトークン流用ではなく、公式 API、公式 CLI、正式な OAuth、正規の接続方法を優先する。
- Obsidian 的な Markdown ベース知識管理、n8n 的な外部連携、VS Code 的 UI、OpenCode 的 CLI/TUI、Zed 的な軽快さなどを参考にしつつ、CommandWeave の一貫した UX に統合する。
- Generative UI 的な返答形式も将来的に取り込み、単なるテキスト応答だけでなくフォーム、表、グラフ、操作パネルで結果を返せるようにする。

- Claude Code、Codex、GitHub Copilot など既存の AI 開発支援ツールを活用可能な構造を持つ。
- それらを競合とみなすのではなく、必要に応じて CommandWeave に接続・補完できる方向を取る。

## 12. Plugin / Skill 審査方針

### 12.1 基本方針

- 攻撃的、破壊的、有害な Plugin / Skill は作成・導入・共有できない。
- 新規 Skill / Plugin は導入前に静的検査と権限確認を行う。
- 外部通信先、必要権限、ファイル操作権限、外部送信有無を明示する。

### 12.2 検査項目

- 危険なコードの有無
- 危険な関数やコマンドの使用
- 外部送信の有無
- 認証情報アクセスの有無
- 削除、更新、公開操作の有無
- 実行対象の範囲
- 使用ライブラリの妥当性
- 同意画面フィッシングや危険な設定誘導の有無
- 秘密情報漏洩の可能性

### 12.3 強制承認対象

以下を含む場合は、人間承認を必須とする。

- 外部送信
- 投稿
- 公開
- 削除
- 課金
- Git push / release
- 重要ファイルの上書き
- 権限変更
- API キー関連処理
- 外部認証情報の更新反映

### 12.4 Skill / Plugin 導入前の実務チェック

- その機能は本体に入れるべき最小核か、Plugin / Extension に切り出すべき拡張か。
- 自然言語から再生成・再構築できる領域か。
- ユーザーが工程追加・削除・API 差し替えを自然言語で指示できるか。
- 共有配布しても安全か、利用規約違反を誘発しないか。

導入前の安全検査では、コード上の危険性だけでなく、実務上の事故も防ぐ必要がある。  
そのため、次の観点も確認対象とする。

- ユーザーに誤認を与える説明になっていないか
- 不要に強い権限を要求していないか
- 本番環境と開発環境を混同していないか
- 外部送信先や保存先が明示されているか
- 更新時に既存データや設定を壊さないか
- 認証情報ローテーション後も正常に動くか

## 13. セキュリティ方針

### 13.1 セントラルリポジトリの考え方

モデル更新情報、料金、特性、推奨用途、注意事項などを毎回個別取得すると、コストや更新負荷が大きくなる。  
そのため CommandWeave では、こうした情報を集約する**セントラルリポジトリ**の考え方を持つ。

このリポジトリは、例えば次の情報を保持する。

- モデル料金
- モデル特性
- 利用可能 Provider
- 推奨用途
- 制約事項
- 更新日
- 既知の不具合や注意点

これにより、各 Skill や Plugin が個別に同じ情報を取得し続けることを避け、API コストと更新負荷を抑える。

### 13.2 参考にする外部ツール・接続先の例

CommandWeave は独自実装に閉じるのではなく、既存の有用なツールや接続方式を必要に応じて取り込む。

参考対象の例:
- LiteLLM
- OpenRouter
- Ollama
- MCP
- Helicone
- Portkey
- n8n
- Claude Code
- Codex
- GitHub Copilot

これらはそのまま採用する対象ではなく、**どの思想・機能・接続性を取り込むか**を見極めたうえで活用する。

### 13.3 セキュリティ方針

- API キーや認証情報は暗号化して管理する。
- 通常ログに平文の機密情報を出力しない。
- 悪意あるウェブサイトや危険な外部リソースへの接続を制限する。
- 使用中の AI サービス、ツール、API の仕様変更、利用規約変更、セキュリティ情報を公式ソースから取得する仕組みを持つ。
- RSS 等による公式情報の定期取得を行えるようにする。
- API キーなどの重要情報を定期的に更新しやすいよう、更新支援用の Skill / Plugin / Extension を用意する。
- 自組織内の安全確認のための防御的セキュリティテストは許可するが、攻撃的用途や第三者対象の挙動は許可しない。
- 追加導入される Skill / Plugin / Extension は、事前に安全性検証を行う。
- Skill のマージ前に静的解析、依存関係確認、秘密情報漏洩検査を行う。
- 会話履歴、改善ログ、共有物に秘密情報の実値を残さない。

## 14. 認証情報ローテーション方針

CommandWeave は、外部 AI サービスやクラウド連携に用いる認証情報について、**自動更新**ではなく**安全なローテーション支援**を基本方針とする。

### 14.1 基本原則

- API キー、OAuth、サービスアカウントなど、認証方式の違いを区別して管理する。
- 期限監視、権限棚卸し、更新提案、反映確認を一連のフローとして扱う。
- 料金、権限、公開設定、本番影響がある変更は必ず人間承認を必要とする。
- 秘密情報の実値は表示、保存、共有を最小化し、原則として Secret Manager や OS 標準の安全な保管先に置く。

### 14.2 想定フロー

1. 接続先ごとに認証方式、権限範囲、作成日、最終確認日、適用環境を台帳化する。  
2. 一定期間ごとに、長期未更新、過剰権限、開発・本番共用などのリスクを検出する。  
3. ユーザーに対して、更新推奨理由、影響範囲、差し替え箇所、ロールバック手順を提示する。  
4. 人間承認後に、新しい認証情報の発行手順や差し替え手順を案内する。  
5. 更新後は疎通確認テストを行い、問題がなければ旧キーの無効化を促す。  

### 14.3 実装方針

- 外部 SaaS の API キーを一律に自動再発行する機能は、初期段階では持たない。
- 自動化する場合も、自社管理下の内部 API や Secret Manager 連携済みの資格情報など、限定領域から始める。
- 本体では、認証情報の棚卸し、期限監視、更新提案、疎通確認、監査ログを中核機能とし、サービス別詳細は Plugin / Extension で拡張する。

### 14.4 想定 Plugin / Skill 例

- Credential Hygiene Plugin
- Secret Rotation Assistant Skill
- Post-Rotation Verification Skill

## 16. 参照思想・参考にする運用原則

CommandWeave は、他ツールの思想をそのまま流用するのではなく、必要な原則のみを取り込む。

### 16.1 参考にする方向性

## 27. まとめ

CommandWeave は、単なる AI チャットや単発自動化ツールではなく、自然言語で業務を構築し、複数 AI を組織として運用し、人間承認と監査可能性を組み込んだ AI オーケストレーション基盤である。

その本質は、AI にすべてを丸投げすることではなく、AI を見える形で組織化し、人間の意志決定と組み合わせながら、長期的に信頼できる業務実行環境を作ることにある。

最初に作るべきものは、すべてを盛り込んだ巨大システムではない。  
自然言語入力、計画提示、複数 AI 実行、Judge、差し戻し、人間承認、履歴保存という核を end-to-end で成立させ、その上に Skills、Plugins、Extensions を積み上げていくことが重要である。

### 27.1 補足まとめ

CommandWeave の価値は、単一の高性能 AI を置くことではなく、**複数 AI を役割分担させ、計画・実行・検証・改善を見える形で回すこと**にある。  
さらにその上で、**人間の承認、監査可能性、権限管理、安全な拡張性**を維持する点に本質がある。

そのため、今後の設計・実装では次を見失わないことが重要である。

- 何を本体に持ち、何を Plugin / Extension に切り出すか
- どこまでを MVP とし、どこから先を将来構想とするか
- 便利さと安全性のどちらも崩さないこと
- AI をブラックボックス化せず、ユーザーが介入できること
- 長期運用できる構造を優先すること


## 28. 重要な要素

### 28.1 利用規約追従の原則

- 外部 AI サービスの利用規約、認証方式、提供範囲は変化しうるため、公式情報を継続的に確認する。
- 非公式なトークン流用や規約違反となりうる接続は避け、公式 API、公式 CLI、正式な OAuth を優先する。
- 規約・料金・プライバシーに関わる判断は、自動実行ではなく計画段階で人間が確認する。

### 28.2 OpenCode / VS Code 的な方向性の継承

- CLI / TUI は軽量で高速、低スペック環境でも扱いやすいことを重視する。
- GUI は非エンジニアでも扱いやすく、拡張管理や状況把握がしやすいことを重視する。
- つまり、OpenCode 的な軽さと VS Code 的な拡張性を両立する方向を取る。

### 28.3 MCP・外部接続・ワークフロー連携

- Google Drive、GitHub、Slack などの外部情報源やツールと安全に接続できることを重視する。
- MCP や iPaaS 的な接続思想を取り込み、外部サービス連携を標準化しやすくする。
- ローカルファイル、クラウドファイル、各種 SaaS を横断して扱えることが望ましい。

### 28.4 内部処理のトークン最適化

- ユーザー向け表示は日本語を維持しつつ、内部表現や中間処理では必要に応じて低トークンな表現を使い、コスト最適化を図る。
- ただし、監査性や説明可能性を損なわないことを前提とする。

### 28.5 24時間運用の考え方

- ユーザーが非アクティブでも継続実行できる業務は長時間運転を許可できるようにする。
- ただし、外部送信、公開、課金、権限変更、重要データ削除は必ず人間承認を要する。

## 29. 並列開発・メタスキルの扱い

### 29.1 並列開発の原則

- Plan を先に詰める。
- AI の行動規範を共通文書で固定する。
- 作業領域を分離し、衝突を防ぐ。
- 最後に統合検証を行う。

これは CommandWeave 自体の開発にも、将来の AI チーム運用にも有効な原則である。

### 29.2 メタスキルの位置づけ

CommandWeave は単にハードスキル的な処理を増やすだけでなく、学習、批判的思考、自己評価、改善提案のようなメタスキル的振る舞いを AI 組織に持たせることを志向する。これにより、単発実行だけでなく、継続運用時の改善余地が大きくなる。

## 29.5 並列実行時の作業分離原則

複数 AI を並列に動かす場合、単に同じ文脈を共有させるだけでは衝突や責務不明瞭が起こりやすい。  
そのため CommandWeave では、並列実行時に少なくとも次を明示する。

- 各担当 AI の責務範囲
- 触れてよい成果物・編集対象
- 依存関係のある前提資料
- レビュー担当と実装担当の分離
- 統合前の確認手順
- 衝突時の優先ルール

将来的には、各 AI に対して**担当領域ごとの作業ブランチ、成果物単位の分離、統合前チェック**を持たせられる構造を取り入れる。  
これは開発作業だけでなく、競合調査班・分析班・文章生成班・レビュー班のような**業務チーム運用**にも適用できる。

## 35. 要望の整理

以下は要望を論点別に整理したものである。

### 35.1 理念・方向性

- システム本体の更新頻度は抑え、最新技術は Skills / Plugins 側で吸収したい。
- AI が AI を指揮するオーケストレーションを実現したい。
- 単機能な AI SaaS ではなく、企業全体の業務をシームレスに扱える基盤にしたい。
- セキュリティ、権限、保守運用まで含めて企業利用に耐える必要がある。
- 企業や個人が、自分たちの現場に特化した AI を容易に作れる環境にしたい。

### 35.2 最重要目的

CommandWeave の最重要目的は、既存 AI エージェントの最大の課題である「リスク管理と信頼構造の欠如」を解消することである。

具体的には、以下に対応できる基盤を目指す。

- 誰がどの AI に何を任せたかを追跡できること
- 過剰権限を避けること
- 操作ログを明確に残すこと
- 退職・担当変更・運用停止時にも制御できること
- ERP / CRM / セキュリティのような慎重さが必要な領域とも接続できること

- spec / plan / tasks の中間成果物管理  
- 自然言語トリガーと内部コマンドモデルの対応づけ  
- 並列 AI の責務分離と衝突防止  
- 改善知識の階層化保存  
- 状態管理、イベント記録、コスト分析の可視化  
- コーディング系 Plugin 向けの意味的コード理解  

### 35.3 参照したい外部思想

CommandWeave は、OpenCode の以下の思想を継承対象とする。

- 軽量・高速
- CLI / TUI ベースでも使いやすい
- 低スペック環境でも動きやすい
- 柔軟なモデル選択
- MCP 対応
- 多数の LLM Provider への接続
- LSP 統合
- セッション共有
- スクリプトとの統合

その上で、OpenCode と異なり、非エンジニアでも扱える GUI を追加し、VS Code のような UI/UX に近づける。

また、OpenClaw 系の運用思想も一部参考にする。特に次の点は CommandWeave と相性がよい。

- 実行担当 AI と監督担当 AI の責務分離
- 改善提案や振り返りを append-only で蓄積する運用
- 自動で設定変更を反映しない慎重な改善フロー
- 秘密情報をメモリや改善ログに残さない原則
- 最小権限を前提とした運用設計

ただし、これはあくまで「自己改善ループ」や「監督 AI の運用原則」の参考であり、CommandWeave 全体のアーキテクチャをそのまま流用するものではない。

### 35.4 UI / 使用感の要望

- GUI は VS Code 風、CLI / TUI は OpenCode 風にする。
- Skills / Plugins を一覧から追加・削除・確認できるようにする。
- チャットから業務用の Skills / Plugins を作成・修正できるようにする。
- 業務状況、進捗、コスト、使用 API をユーザーが常に確認できるようにする。
- AI が API を自動選定しても、ユーザーがコスト重視・品質重視・無料枠重視を選べるようにする。
- ローカル・クラウド・各種サービスに広く接続できるようにする。

### 35.5 セキュリティ・懸念点

- 最低限どの機能を本体に入れ、何を Plugin 化するかの基準を明確にする必要がある。
- Skill / Plugin をチャットで生成できても、安全性検証なしには導入させない。
- 大規模システムの置き換えでは、要件充足と保守運用の証明が必要になる。
- 同意画面フィッシングや危険な設定ファイルを含むリポジトリの取り扱いに注意する。
- Skill マージ前に静的解析や秘密情報漏洩チェックを入れる。
- OAuth は初期設定で必須にせず、必要に応じて Extension で有効化する方向も検討する。
- API キーやトークンの実値は、会話履歴・改善ログ・共有用 Skill / Plugin に残さない。
- 外部サービスの認証情報は、無条件に自動更新・自動反映しない。
- 認証情報の更新は、人間承認を前提とした安全なローテーション支援として設計する。

### 35.6 認証情報ローテーション方針

CommandWeave は、外部 AI サービスやクラウド連携に用いる認証情報について、「自動更新」ではなく「安全なローテーション支援」を基本方針とする。

#### 基本原則

- API キー、OAuth、サービスアカウントなど、認証方式の違いを区別して管理する。
- 期限監視、権限棚卸し、更新提案、反映確認を一連のフローとして扱う。
- 料金、権限、公開設定、本番影響がある変更は必ず人間承認を必要とする。
- 秘密情報の実値は表示・保存・共有を最小化し、原則として Secret Manager や OS 標準の安全な保管先に置く。

#### 想定フロー

1. 接続先ごとに認証方式、権限範囲、作成日、最終確認日、適用環境を台帳化する。  
2. 一定期間ごとに、長期未更新、過剰権限、開発・本番共用などのリスクを検出する。  
3. ユーザーに対して、更新推奨理由、影響範囲、差し替え箇所、ロールバック手順を提示する。  
4. 人間承認後に、新しい認証情報の発行手順や差し替え手順を案内する。  
5. 更新後は疎通確認テストを行い、問題がなければ旧キーの無効化を促す。

#### 実装方針

- 外部 SaaS の API キーを一律に自動再発行する機能は、初期段階では持たない。
- 自動化する場合も、自社管理下の内部 API や Secret Manager 連携済みの資格情報など、限定領域から始める。
- CommandWeave 本体では、認証情報の棚卸し、期限監視、更新提案、疎通確認、監査ログを中核機能とし、サービス別の詳細フローは Plugin / Extension で拡張する。

#### 想定 Skill / Plugin 例

- Credential Hygiene Plugin
- Secret Rotation Assistant Skill
- Post-Rotation Verification Skill

### 35.7 AI エージェント運用の要望

- Agent Teams 型のマルチ AI 組織を基本とする。
- 役割ごとに班や部署を分け、双方向通信や共有タスクリストで連携できるようにする。
- AI 秘書、SNS 運用、リモート操作、ブログ・動画・音声の横展開などの高頻度業務を Plugin 化したい。
- ユーザーの価値観や判断傾向を学習して、分身 AI として育つ方向を目指したい。
- 常時稼働・長期記憶・提案型の AI 組織を実現したい。

### 35.8 ログイン・接続の要望

- OpenRouter 固定ではなく、専用アカウントや Google アカウントなど柔軟なログイン方式を持たせたい。
- OpenRouter 以外の AI API アグリゲーターやローカル LLM にも対応したい。
- Claude Code、Codex、GitHub Copilot などの既存サービスを有効活用したい。
- ローカル LLM や無料枠を活用し、API コストゼロ運用も選べるようにしたい。

### 35.9 将来方針

- 無料・軽量・高機能な AI エージェント版 VS Code のような位置づけを目指す。
- コーディングに限らず、知識作業や現場業務にも対応する。
- 将来的には AI ロボット、スマートグラス、AR / VR、ハードウェア開発にも転用したい。
- 法律、税務、経済など専門ナレッジを扱う部署型 AI も視野に入れる。

### 35.10 メタスキルの導入

現在の AI エージェントには、ハードスキルやソフトスキルはあっても、学び方・見通し方・自己修正のようなメタスキルが弱い。  
CommandWeave では、AI 組織に以下の能力を持たせたい。

- 学習能力
- 批判的思考
- 自己認識
- 好奇心
- システム思考
- 実行と改善の往復能力

### 35.11 並列開発への示唆

Claude Code の並列開発事例から、次の方針を参考にする。

- 実装前の計画に十分な時間をかける
- AI の行動ルールを `CLAUDE.md` などで明文化する
- `git worktree` などを使って並列実装を衝突なく進める
- テストや検証は Skill 化して再利用可能にする

### 35.12 Plugin 戦略

CommandWeave は、最初からすべてを詰め込むのではなく、最低限の中核機能の上に必要な Plugin を追加して広げる構造を取る。

初期状態では以下が重要になる。

- 汎用 Skill 作成能力
- Plugin の追加・削除・共有・公開
- Skill / Plugin の安全性検査
- 公式 Plugin のいくつかの同梱
- ユーザーごとの最適化余地の確保
