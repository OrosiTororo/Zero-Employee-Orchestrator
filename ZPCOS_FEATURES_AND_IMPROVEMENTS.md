注）下記のシステム機能でSkillsのまとまりであるプラグインの存在を記していないのと、プラグインやSkillsの定義が曖昧でした。「SkillsをVSCodeの拡張機能のように追加可能」ではなく、「既存のSkillsや新規で作成したSkillsを複数組み合わせたプラグインをVSCodeの拡張機能のように追加可能」という感じです。そのためYouTube SkillsではなくYouTube pluginが正しいです。システムの方でも修正をお願いします。なお、チャットによる自然言語のみでSkillsやそのまとまりであるplaginを作成可能であるかの確認をお願いします。既存のSkillsも最新のものを取得するようにしてください。

## 1. プロジェクト概要（本格的な概要は後述の目的の項に記述済み）

ZPCOS は「AI を道具から組織に進化させる」ローカル常駐型デスクトップ OSである。
テストとしてはYouTube チャンネル運営における、トレンド分析・台本作成・競合調査・パフォーマンス分析などの
「作業」を AI 組織に自律的に任せ、クリエイティブな表現に集中することが可能であることを示している。
が、このシステムの目的は会社のあらゆる業務の自律化を行い、指示・承認を行う社長のみが会社を運営できるようにするためのOSである。
正に「ゼロ・パーソン・カンパニー」を実現するためのOSなので、ZPCOSという名称なのである。
YouTubeのSkillsはあくまで、 SNS分析という複雑なワークフローを実行できるかの確認である。
既存のAIエージェントでも業務を遂行することは可能だが、複数AI監視、自然言語での業務Skillsの作成・共有、出力失敗時に自律的にDAG構築、成功体験、業務同士の繋がりを一つのアプリケーションで行うからこそ、OSという名前がついている。
**既存のAIエージェントのような単発タスクの自動化やマルチステップの遂行、長期的意思決定 /自律計画修正、自律的環境センサー + イベントトリガーだけでなく、マルチエージェント協調・統制や高度な自己改善・反省ループ、自然言語のみで業務を遂行するためのSkillsを自動で組み合わせ・構築してプラグインを作成し、ユーザーの任意で共有・公開が可能（個人情報や機密は共有せずプライバシーが保たれるようにする）である。**

**技術スタック:**
- バックエンド: Python 3.12 / FastAPI / uvicorn / LiteLLM
- フロントエンド: React 19 + TypeScript + Vite + shadcn/ui
- デスクトップ: Tauri v2（Rust）+ PyInstaller サイドカー
- LLM プロバイダー: OpenRouter 経由（6モデルグループ: fast, think, quality, free, reason, value）

## 2. 現在の機能一覧（9層アーキテクチャ）

### Layer 1: User Layer
- 自然言語で目的を入力するだけで AI 組織が動く

### Layer 2: Design Interview（壁打ち・すり合わせ）
- `POST /api/interview/start` — ユーザー入力から質問リスト自動生成
- `POST /api/interview/respond` — 選択式＋自由記述で深掘り
- `POST /api/interview/finalize` — Spec（要件定義書）自動生成
- ユーザーが言語化していない前提を掘り起こす深い質問

### Layer 3: Task Orchestrator（司令塔）
- **Plan/DAG 提案**: タスク分解→Skill 割当→依存関係 DAG を生成→ユーザー承認後に実行
- **Cost Guard**: API コスト事前見積り、予算超過時に代替 Plan 自動提案
- **Quality SLA**: 3段階（FASTEST / BALANCED / HIGH_QUALITY）の品質モード選択
- **Re-Propose**: 差し戻し時の Change Request ベース再提案、Plan Diff 表示
- **Self-Healing DAG** (v11.2): 失敗時に AI が自律的に DAG を再構築
  - 4戦略: RETRY_SAME / SWAP_SKILL / REPLAN / DECOMPOSE
  - 最大3回リトライ、超過で人間にエスカレーション
  - `POST /api/orchestrate/{id}/self-heal`, `GET /api/orchestrate/{id}/heal-history`

### Layer 4: Skill Layer
- **Skill フレームワーク**: SKILL.json + executor.py の2ファイル構成
- **Skill Gap Negotiation**: 不足 Skill 検出→3案提示（代替 / 自動生成 / スキップ）
- **Skill ROI Explainer**: Skill の代替案・価値・リスクを説明
- **Skill 自動生成エンジン**: 自然言語記述から SKILL.json + executor.py を LLM で生成
- **ビルトイン YouTube Skills（5種）**:
  - `yt-script`: 台本生成（フック→本編→CTA の構成付き）
  - `yt-rival`: 競合チャンネル分析（YouTube Data API 連携）
  - `yt-trend`: トレンド分析（急上昇動画・検索トレンド）
  - `yt-performance`: パフォーマンス分析（YouTube Analytics API）
  - `yt-next-move`: 次の一手提案（全 Skill の成果物を統合分析）
- **Local Context Skill** (v11.2): ローカルファイルをセキュアに読み込み AI 分析
  - 対応: .txt, .md, .csv, .pdf, .docx, .png, .jpg
  - 許可ディレクトリ制限、外部送信時はユーザー承認必須

### Layer 5: Judge Layer
- **Two-stage Detection**:
  - Stage 1（ルールベース・安価）: 入力不足 / 禁止事項 / コスト超過 / 危険操作
  - Stage 2（Cross-Model Judge・高精度）: 論理 / 根拠 / 矛盾検証
- **Policy Pack**: カテゴリ別コンプライアンスルール、提案段階で違反を提示

### Layer 6: Re-Propose Layer
- 再実行粒度: FULL_REGENERATE / FROM_STEP_N / PLAN_MODIFY
- Dynamic DAG Rebuild（v11.2）: Judge 差し戻し時の自動再計画

### Layer 7: State & Memory
- **状態機械**: draft → ai_executing → ai_completed → judging → human_review → approved → committed
- **Experience Memory**: 過去の成功体験を aiosqlite に永続化、Plan 生成時に自動参照
- **Failure Taxonomy**: エラー分類（AUTH_ERROR, RATE_LIMIT, SPEC_CHANGE 等）+ 回復戦略提案
- **Artifact Bridge**: 成果物スロット化（insight / copy / data / analysis）、別業務への自動差し込み
- **Knowledge Refresh**: 外部ソース差分取得→要約・重要度判定（MVP は手動トリガー）

### Layer 8: Provider Interface
- **LiteLLM Gateway**: OpenRouter 経由で全モデルにアクセス（openrouter/ プレフィックス）
- **Model Catalog Auto-Update**: OpenRouter API からカタログ自動取得→スモークベンチ→適用→失敗時ロールバック
- **Recommendation Ladder**: AI 内部の「今すぐやる / 次にやる / やらないこと」思考フレーム

### Layer 9: Skill Registry（v11.2）
- コミュニティ Skill の公開・検索・インストール
- パッケージ形式: SKILL.json + executor.py + README.md を ZIP 化
- `GET /api/registry/search`, `POST /api/registry/publish`, `POST /api/registry/install`, `GET /api/registry/popular`

### 認証・セキュリティ
- **Token Store**: keyring（AES-256 鍵）+ AES-GCM 暗号化ファイル
- **OpenRouter OAuth PKCE**: ポート 3000 一時サーバー
- **Google OAuth**: InstalledAppFlow + PKCE（YouTube/Gmail/Calendar/Drive）
- **AuthHub**: 統合認証マネージャー
- **Skill セキュリティ**: インポートホワイトリスト / 関数ブラックリスト（eval, exec, subprocess 等禁止）

### フロントエンド（7画面）
- `/login` — ログイン
- `/interview` — Design Interview（壁打ち画面）
- `/` — ダッシュボード
- `/orchestrate/:id` — オーケストレーション詳細 + Self-Healing 履歴パネル
- `/skill/:name` — Skill 実行
- `/skill/create` — Skill 作成
- `/settings` — 設定

### API エンドポイント（合計 33 個）
- 認証: 7 / Interview: 3 / Orchestrate: 8 / コア: 2 / タスク: 3 / Skill: 4 / Registry: 4 / その他: 2

## 要望

### 理念

- このシステムはAIなどの技術の進歩に柔軟に対応でき、かつ使いやすさを保つためにも、ある程度形や機能が固まったらシステム自体のアップデートは最小限に、最新技術をSkillsだけで追加できるようにしたい。
- このシステムのSkillsやプラグイン作成における組み合わせ・構築機能によって革新的な技術や業務遂行が生まれるのはロマンを感じる。そしてなにより重要なのは、企業が大きく抱えている問題であるAIエージェントの問題を解決できることである。

### 目的（最重要）

**このOSの目的は既存のAIエージェントの問題である「リスク」に企業が対応できるようにするためである。AIが自律的に行動する時代には「誰が、どのAIに、どの範囲を委ねたのか」という信頼の構造設計が不可欠だ。過剰権限の付与、退職後も稼働し続けるAI、操作ログの不透明さ。放置すれば重大な事故につながりかねない。だからこそこのOSはAIを「統制の対象」から「信頼のパートナー」へ変えることを目的としている。また業務プロセスを変革したERPや、顧客データを蓄積したCRM、セキュリティ領域といったAIエージェントとの連携強化、業務プロセスを変革したERPや、顧客データを蓄積したCRM、セキュリティ領域といったAIエージェントに大替しずらい分野にもAIが対応出来るようにもできる正に「ゼロ・パーソン・カンパニー」を実現するためのOSなのだ。AIエージェントとの連携強化で企業の形を変えることを目的としている。個人が一企業分を動かせる新時代のOS。**

### UIや使用について

- SkillsをVSCodeの拡張機能のように追加可能であり、各Skillsやプラグインの内容を確認でき、そこから機能の追加と削除ができる。
- 全体的にVSCodeのように使用可能。プロンプトから業務遂行のためのSkills・プラグインの作成と、業務の状況や追加削除ができる。
- ユーザーの目に見えて業務などの状況が確認できる。プラグインにおける各Skillsの業務や工程で使用するAPIはAIが自動で最適なものを選定するが、ユーザーが指定したり、コストを抑えるか、質重視か、無料分のみで行うかを指示することも可能。
- もちろん共有や公開されたSkillsやプラグインでもAPIの指定や業務の追加削除がプロンプトで自然言語で指示するだけで可能。

### 懸念点

- どの位までアプリケーションに入れるか、プラグインにするかが重要であり、最低限の機能は最初から組み込まれていなければならない。
- 幅広い言語に対応する為に、言語設定Skillsも必要なのか。アプリケーション本体の言語はインストール時に設定した方がいいのか。
- 自分でスキル（カスタムワークフロー）を作成するなどは、AIエージェントでも可能であるため、既存のツールと差別化や優位性があることを証明できるのか。
- どの位までをシステムに最初から入れるのか、プラグインにするかが重要であり、誰でも使えるものでなくてはならない。
- YouTube プラグインが最初からシステムに入れる必要は無いが、公式が用意したプラグインを幾つか作成しておいて、まだユーザー間でのSkills・プラグイン共有、公開が殆どない状態でも、ある程度の拡張性を作れる。
- Skillsやプラグインを追加できるといってもチャットで作成可能であるのと、追加したプラグインの工程を追加、削除するようにSkillsやAPIを変更できて、AIがAPI料金や組織間で外部・内部の情報と状況を把握しているという利点があるが、基本的なシステムはこれでいいのか、後の追加要素はプラグインで対応させるのかが曖昧。
- 大規模システムの置き換えの場合、データベース設計や要件を満たしているかの確認も必要である。保守運用できる要素を証明する必要がある。

### ログイン時やカスタマイズについて

- ログイン時にOpenRouterのアカウントを求めるのでは無く、専用のアカウントやGoogleアカウントなどを求めた方がいい。OpenRouterのAPIサービスに限定するのでは無く、他のAI APIアグリゲーター（Together AIやHyperbolicなど）にも対応したり、ユーザーがAPIキーを設定したい場合もあるので、OpenRouterに固定せずに行いたい。
- AIやAPI、ツールといったサービスもユーザーが選べるようカスタマイズ性を高める。

### 将来的な方針について

VSCodeのような無料・軽量かつ高機能なオープンソースのコードエディターのAIエージェント版にするのもいいが、ビジネスに利用する場合はSaaSのようにするか、API料金の一部を手数料として貰うのかなどの方向性があるが、起業最初に公表するシステムとしてはどの様な形にするか決める必要があるが決め切れないため、システムの運用として最も適した形にしてほしい。

### プラグインについて

ZPCOSにおいて、ユーザーが最初にインストールした段階ではSkillsは最低限しか無い状態であり、ユーザーがチャットでAIと対話し合い、AIに行わせたい業務を遂行するためのSkillsを組み合わせたり、新しく構築しまとめてプラグインにする。また、YouTube pluginのように公式や他のユーザーが作成した業務遂行プラグインを追加することもできるプラットフォームでもある。正にAI版VCodeのように拡張機能のような形でカスタマイズできるのだ。最初から詰め込むのではなく、必要なものを必要な分だけ取り入れることで、どんな環境にも適応、応用できる自由度の高い拡張的で柔軟なシステムとなる。
