# ZPCOS 機能一覧・改善案・要望まとめ

> 作成日: 2026-03-03
> プロジェクト名: Zero-PCOS（Zero-Person Campany Orchestration System）

---

注）下記の機能一覧や改善案でSkillsのまとまりであるプラグインの存在を記していないのと、プラグインやSkillsの定義が曖昧でした。SkillsはAIに教える具体的な「手順や知識」であり、Pluginはそれらを配布・インストールしやすくまとめた「コンテナ（詰め合わせ）」です。 改善案よりも要望の方がアイデアの質は高いです。
「SkillsをVSCodeの拡張機能のように追加可能」ではなく、「既存のSkillsや新規で作成したSkillsを複数組み合わせたプラグインをVSCodeの拡張機能のように追加可能」という感じです。そのためYouTube SkillsではなくYouTube pluginが正しいです。システムの方でも修正をお願いします。なお、チャットによる自然言語のみでSkillsやそのまとまりであるplaginsを作成可能であるかの確認をお願いします。既存のSkillsも最新のものを取得するようにしてください。

## 1. プロジェクト概要（本格的な概要は後述の目的の項に記述済み）

- Zero-PCOS はAIの進歩に柔軟に対応できる次世代AI統合開発環境（IDE）であり、VSCodeやGoogle AntigravityといったコードエディタのAIエージェント版である。
- 「AI を道具から組織に進化させる」ローカル常駐型デスクトップとしても仮想サーバー上でも動くOrchestration Systemである。
- 既存のAIツールやワークフロー、APIに幅広く対応できる。
- テストとしてはYouTube チャンネル運営における、トレンド分析・台本作成・競合調査・パフォーマンス分析などの「作業」を AI 組織に自律的に任せ、クリエイティブな表現に集中することが可能であることを示している。
- このシステムの目的は会社のあらゆる業務の自律化を行い、指示・承認を行う1名のみで会社を運営できるようにするためのOSであり正に「ゼロ・パーソン・カンパニー」を実現するためのOSなので、Zero-PCOSという名称なのである。
- AIエージェントがどのようなプロセスで判断・行動したかが見えるため、AIエージェントの「ブラックボックス」を無くす事が可能である。
- YouTube Plaginはあくまで、 SNS分析という複雑なワークフローを実行できるかの確認である。
- 既存のAIエージェントでも業務を遂行することは可能だが、複数AI監視、自然言語での業務Skillsの作成・共有、出力失敗時に自律的にDAG構築、成功体験、業務同士の繋がりを一つのアプリケーションで行うからこそ、OSという名前がついている。
- 複数のモデルを統合する（LiteLLM Proxyなど）
OpenAIのAPI形式のままで、Anthropic (Claude) や Google (Gemini)、またはローカルモデル (Ollama) を呼び出せるようにします。これによりコードを書き換えることなく、環境変数（Base URL）を変更するだけでモデルを切り替えられます。
- GitHub Copilot などの通信を中継する
社内LANやVPN環境で、プロキシサーバーを介さないとAIツールが通信できない場合に設定します。ただしSSL証明書の検証エラー（Self-signed certificate）が起きやすいため、証明書の設定が必要になることが多いため、そこを解決してください。
- APIコストの管理・キャッシュにより、チーム開発で、同じプロンプトに対する回答をキャッシュしてAPI費用を節約したり、ユーザーごとの利用制限（クォータ）を設定したりします。ツール例としてはHelicone や Portkey。これらは「AI Gateway」とも呼ばれ、APIキーを隠蔽しつつ利用状況を可視化できます。以下の四つのサイトを参照。
- https://github.com/Helicone
- https://github.com/Helicone/helicone
- https://github.com/portkey-ai
- https://github.com/stardog-union/portkey-gateway
- 現場での全自動化や長期運用を見通す保守の見識に足るAIを運営する事が第一である。
- **社外秘データや個人情報がAIの学習に使われたり、外部に漏洩しない。**
- **既存のAIエージェントのような単発タスクの自動化やマルチステップの遂行、長期的意思決定 /自律計画修正、自律的環境センサー + イベントトリガーだけでなく、マルチエージェント協調・統制や高度な自己改善・反省ループ、自然言語のみで業務を遂行するためのSkillsを自動で組み合わせ・構築してプラグインを作成し、ユーザーの任意で共有・公開が可能（個人情報や機密は共有せずプライバシーが保たれるようにする）である。**
- **必ず上記や下記の内容が実装されているな確認すること。**

**技術スタック:**
- バックエンド: Python 3.12 / FastAPI / uvicorn / LiteLLM
- フロントエンド: React 19 + TypeScript + Vite + shadcn/ui
- デスクトップ: Tauri v2（Rust）+ PyInstaller サイドカー
- LLM プロバイダー: OpenRouter 経由（6モデルグループ: fast, think, quality, free, reason, value）

## 2. 現在の機能一覧（9層アーキテクチャ）

### Layer 1: User Layer
- 自然言語で目的を入力するだけで AI 組織が動く

### Layer 2: Design Interview（壁打ち・すり合わせ）
- `POST /api/interview/start` — ユーザー入力から質問リスト自動生成
- `POST /api/interview/respond` — 選択式＋自由記述で深掘り
- `POST /api/interview/finalize` — Spec（要件定義書）自動生成
- ユーザーが言語化していない前提を掘り起こす深い質問

### Layer 3: Task Orchestrator（司令塔）
- **Plan/DAG 提案**: タスク分解→Skill 割当→依存関係 DAG を生成→ユーザー承認後に実行
- **Cost Guard**: API コスト事前見積り、予算超過時に代替 Plan 自動提案
- **Quality SLA**: 3段階（FASTEST / BALANCED / HIGH_QUALITY）の品質モード選択
- **Re-Propose**: 差し戻し時の Change Request ベース再提案、Plan Diff 表示
- **Self-Healing DAG** : 失敗時に AI が自律的に DAG を再構築
  - 4戦略: RETRY_SAME / SWAP_SKILL / REPLAN / DECOMPOSE
  - 最大3回リトライ、超過で人間にエスカレーション
  - `POST /api/orchestrate/{id}/self-heal`, `GET /api/orchestrate/{id}/heal-history`

### Layer 4: Skill Layer
- **Skill フレームワーク**: SKILL.json + executor.py の2ファイル構成
- **Skill Gap Negotiation**: 不足 Skill 検出→3案提示（代替 / 自動生成 / スキップ）
- **Skill ROI Explainer**: Skill の代替案・価値・リスクを説明
- **Skill 自動生成エンジン**: 自然言語記述から SKILL.json + executor.py を LLM で生成
- **ビルトイン YouTube Skills（5種）**:
  - `yt-script`: 台本生成（フック→本編→CTA の構成付き）
  - `yt-rival`: 競合チャンネル分析（YouTube Data API 連携）
  - `yt-trend`: トレンド分析（急上昇動画・検索トレンド）
  - `yt-performance`: パフォーマンス分析（YouTube Analytics API）
  - `yt-next-move`: 次の一手提案（全 Skill の成果物を統合分析）
- **Local Context Skill** : ローカルファイルをセキュアに読み込み AI 分析
  - 対応: .txt, .md, .csv, .pdf, .docx, .png, .jpg
  - 許可ディレクトリ制限、外部送信時はユーザー承認必須

### Layer 5: Judge Layer
- **Two-stage Detection**:
  - Stage 1（ルールベース・安価）: 入力不足 / 禁止事項 / コスト超過 / 危険操作
  - Stage 2（Cross-Model Judge・高精度）: 論理 / 根拠 / 矛盾検証
- **Policy Pack**: カテゴリ別コンプライアンスルール、提案段階で違反を提示

### Layer 6: Re-Propose Layer
- 再実行粒度: FULL_REGENERATE / FROM_STEP_N / PLAN_MODIFY
- Dynamic DAG Rebuild: Judge 差し戻し時の自動再計画

### Layer 7: State & Memory
- **状態機械**: draft → ai_executing → ai_completed → judging → human_review → approved → committed
- **Experience Memory**: 過去の成功体験を aiosqlite に永続化、Plan 生成時に自動参照
- **Failure Taxonomy**: エラー分類（AUTH_ERROR, RATE_LIMIT, SPEC_CHANGE 等）+ 回復戦略提案
- **Artifact Bridge**: 成果物スロット化（insight / copy / data / analysis）、別業務への自動差し込み
- **Knowledge Refresh**: 外部ソース差分取得→要約・重要度判定（MVP は手動トリガー）

### Layer 8: Provider Interface
- **LiteLLM Gateway**: OpenRouter 経由で全モデルにアクセス（openrouter/ プレフィックス）
- **Model Catalog Auto-Update**: OpenRouter API からカタログ自動取得→スモークベンチ→適用→失敗時ロールバック
- **Recommendation Ladder**: AI 内部の「今すぐやる / 次にやる / やらないこと」思考フレーム

### Layer 9: Skill Registry
- コミュニティ Skill の公開・検索・インストール
- パッケージ形式: SKILL.json + executor.py + README.md を ZIP 化
- `GET /api/registry/search`, `POST /api/registry/publish`, `POST /api/registry/install`, `GET /api/registry/popular`

### 認証・セキュリティ
- **Token Store**: keyring（AES-256 鍵）+ AES-GCM 暗号化ファイル
- **OpenRouter OAuth PKCE**: ポート 3000 一時サーバー
- **Google OAuth**: InstalledAppFlow + PKCE（YouTube/Gmail/Calendar/Drive）
- **AuthHub**: 統合認証マネージャー
- **Skill セキュリティ**: インポートホワイトリスト / 関数ブラックリスト（eval, exec, subprocess 等禁止）

### フロントエンド（7画面）
- `/login` — ログイン
- `/interview` — Design Interview（壁打ち画面）
- `/` — ダッシュボード
- `/orchestrate/:id` — オーケストレーション詳細 + Self-Healing 履歴パネル
- `/skill/:name` — Skill 実行
- `/skill/create` — Skill 作成
- `/settings` — 設定

### API エンドポイント（合計 33 個）
- 認証: 7 / Interview: 3 / Orchestrate: 8 / コア: 2 / タスク: 3 / Skill: 4 / Registry: 4 / その他: 2

## 要望

### 理念

- このシステムはAIなどの技術の進歩に柔軟に対応でき、かつ使いやすさを保つためにも、ある程度形や機能が固まったらシステム自体のアップデートは最小限に、最新技術をSkillsだけで追加できるようにしたい。
- このシステムのSkillsやプラグイン作成における組み合わせ・構築機能によって革新的な技術や業務遂行が生まれるのはロマンを感じる。そしてなにより重要なのは、企業が大きく抱えている問題であるAIエージェントの問題を解決できることである。
- **業務用SaaSのイチ機能だけ切り出してそれっぽい機能をAIでつくることは容易になりつつありますが、基幹システムといわれる様なSaaSはそのイチ機能が何百、何千もあるイメージです。そのあらゆる機能をシームレスに動かし、さらにセキュリティや権限設定まで作り込み、それを毎日保守運用していくというのは「イチ機能がAIで簡単につくれる」のとは天と地の差があります。このZPCOSはその従来のAIにおける課題を解決するためにあり、SaaSからAIへの完全大替と企業まるごとAI化を目指したOSです。**
- セキュリティやプライバシーなどの安全面の確保といった信頼性が最優先。
- 用途特化（バーティカル）のAIアプリケーションを誰でも容易に開発可能にする。
- 現場でのAI開発を活発化し、専門分野に寄ったAI、企業ごとの強みを活かしたAIが多く生まれることを願う。

### 目的（最重要）

**このOSの目的は既存のAIエージェントの問題である「リスク」に企業が対応できるようにするためである。AIが自律的に行動する時代には「誰が、どのAIに、どの範囲を委ねたのか」という信頼の構造設計が不可欠だ。過剰権限の付与、退職後も稼働し続けるAI、操作ログの不透明さ。放置すれば重大な事故につながりかねない。だからこそこのOSはAIを「統制の対象」から「信頼のパートナー」へ変えることを目的としている。また業務プロセスを変革したERPや、顧客データを蓄積したCRM、セキュリティ領域といったAIエージェントとの連携強化、業務プロセスを変革したERPや、顧客データを蓄積したCRM、セキュリティ領域といったAIエージェントに大替しずらい分野にもAIが対応出来るようにもできる正に「ゼロ・パーソン・カンパニー」を実現するためのOSなのだ。AIエージェントとの連携強化で企業の形を変えることを目的としている。個人が一企業分を動かせる新時代のOS。**

### 懸念点

- どの位までアプリケーションに入れるか、プラグインにするかが重要であり、最低限の機能は最初から組み込まれていなければならない。
- 幅広い言語に対応する為に、言語設定Skillsも必要なのか。アプリケーション本体の言語はインストール時に設定した方がいいのか。
- 自分でスキル（カスタムワークフロー）を作成するなどは、AIエージェントでも可能であるため、既存のツールと差別化や優位性があることを証明できるのか。
- どの位までをシステムに最初から入れるのか、プラグインにするかが重要であり、誰でも使えるものでなくてはならない。
- YouTube プラグインが最初からシステムに入れる必要は無いが、公式が用意したプラグインを幾つか作成しておいて、まだユーザー間でのSkills・プラグイン共有、公開が殆どない状態でも、ある程度の拡張性を作れる。
- Skillsやプラグインを追加できるといってもチャットで作成可能であるのと、追加したプラグインの工程を追加、削除するようにSkillsやAPIを変更できて、AIがAPI料金や組織間で外部・内部の情報と状況を把握しているという利点があるが、基本的なシステムはこれでいいのか、後の追加要素はプラグインで対応させるのかが曖昧。
- 大規模システムの置き換えの場合、データベース設計や要件を満たしているかの確認も必要である。保守運用できる要素を証明する必要がある。
- YouTube plaginに関しては自然言語でのチャットで構築できるか確認したいので、一度消去し構築できるのかテストを行いたい。
- ユーザー側がAIにどこまで許可するかを明確にするために、計画時に確認事項をまとめ、提示した方がいい。
- セキュリティソフトの導入やデータ漏洩対策はもちろんのこと、銀行やデータセンサーのセキュリティを任せられるほどのハッカー・ウイルス対策と対処をAI組織で確立し、日々アップデートして最新の脅威に備える。
- **APIについて散々書いてきましたが、AIプロキシを活用し、複数のモデルを統合する（LiteLLM Proxyなど）は、OpenAIのAPI形式のままで、Anthropic (Claude) や Google (Gemini)、またはローカルモデル (Ollama) を呼び出せるようにできます。メリット: コードを書き換えることなく、環境変数（Base URL）を変更するだけでモデルを切り替えられます。活用例: Cursor や VSCode (Continue) 及びZero-PCOSの接続先を自前プロキシに向けることで、低コストのモデルや最新モデルを使用できます。**下の二つのサイトを参照
- https://github.com/BerriAI/liteLLM-proxy
- https://github.com/BerriAI/litellm
- AIエージェントにおける脆弱性は「設定ファイルがそのまま攻撃ベクターになる」という構造的な問題だ。従来の開発では、.gitignoreやpackage.jsonを「ただのメタデータ」として扱ってきた。コードレビューでも設定ファイルの変更は流し見されがちだ。しかしAIエージェントの登場で、設定ファイルは「実行ロジックの定義」に変貌した。これはAI開発ツールが増えるほど、「設定ファイル＝信頼できるメタデータ」という前提は崩壊していく。**最も重要な教訓: 信頼できないリポジトリは、AI開発ツールで開く前に設定ファイルを必ず確認すること。**

### AIエージェントについて（重要）

- Zero-PCOSにおけるAIエージェントはClaude Code Agent Teamsのように複数の独立したAIエージェントを「チーム」として組織し、複雑なタスクを並列かつ自律的に解決させることができるエージェントチーム (Agent Teams)で構成されます。
- サブエージェント (Subagents)は通信がリーダーへの報告のみ（一方通行）、調整がリーダーがすべて管理、ユーザー操作がリーダー経由のみ介入可能、適したタスクが独立した単発の調査・修正のみ。
- エージェントチーム (Agent Teams)は通信がメンバー間での双方向通信が可能、調整が共有タスクリストで自律的に調整、	ユーザー操作が個別のメンバーに直接指示が可能、適したタスクが依存関係のある複雑な並列開発で可能です。
- 従来のサブエージェントが「特定タスクをこなして報告する部下」であるのに対し、エージェントチームは「Slackのように会話しながら並行作業するプロジェクトメンバー」です。
- **DiscordやLINE、Slackといった普段使っているメッセージアプリから、家のパソコンにある全てのアプリをリモコンみたいに動かせるAI秘書機能をプラグインで追加できるようにしたい。「今日の予定」や「昨日の残タスク」を自動送信してくれたり、ブリーフィングを送ってくれて、カレンダーの認証エラーが出たら自分で直そうとするし、もう「指示待ち」じゃなくて完全に自律して動いてくれて日々成長する。またユーザーが食べたものや運動記録なども共有すれば管理パートナー化もでき、「ここが欠点だから私がカバーするね」のような提案をしてくれたり、「今は意外と余裕があるから寝ても大丈夫だよ」のように休憩の指示までくれるイメージ。ユーザーが休むべきか働くべきか分からなくなる時にこういう一言でQOLが劇的に上がる。**
- OpenClawのようにWhatsAppやSlack、Discordなど数多くのチャットアプリからAIに「実際に行動させる」パーソナルエージェントとしても機能できるようにする。Open Clawについては以下を参照。
- https://github.com/openclaw/openclaw
- **Claude Code：システムやワークフロー自体を開発するエンジニア　OpenClaw：開発されたシステムを使って日々の業務を回すワーカー仮想サーバー上に両方を立てておくと、AI同士が勝手に会話して、私が寝ている間にもシステムをどんどんカスタムして最適化してくれるため、Zero-PCOSでも仮想サーバー上でこの機能を使えるようにし、この2種類のAI以外にも多くのAIが組織になって業務や自動化を行う。複数の部署に分かれ業務や工程ごとに班が分かれたAIエージェントが全自動で仕事するAI組織を作る。正にゼロパーソンカンパニー。**
- **収入が不安定なフリーランスこそAI秘書が必要であると考えている。「作る系の単発仕事が終わると、急にニート状態になる」っていう不安定さがあるからだ。だからこそ、「来月ちょっと仕事減ってるよ？ 今のうちに動いた方がいいんじゃない？」のように先回りしてアラートを出してくれる機能や、普段から「今日何やった」「これ作った」っていう事実を全部AIに共有して、自分専属のマネージャーとして育てていく。それもZero-PCOS上であれば一つのアプリケーション上で一括で可能にする。**
- **AIにSNSを任せると内容が事実とズレたりしがちだが、毎日新鮮な「事実」という裏付けされた情報を渡しておけば、そこから最高の投稿を仕上げてくれる。普段から情報を共有して、AIにSNSを任せれるようにすること。**
- **AI秘書と連携してブログを書いたら、そこからYouTubeやXなどの他のSNS媒体の投稿も作成したり、動画生成や、音楽生成などの応用も自動で行う。もちろん修正や投稿するかどうかのチェックは必ず人間であるユーザーのチェックが必要。ユーザーからのネタという「素材」が1個あれば、それを薄めたり、形を変えたり、伸ばしたりして無限にコンテンツが作れるイメージ。**
- PCを「丸ごと遠隔操作」する仕組みだから、悪意のあるプラグインを入れたり設定を間違えると、ローカルの情報を全部持っていかれるリスクがあるため、そこら辺の管理を行うAIエージェントの部署も必要。
- AIエージェントに不足しているのは人間味であり、体験談、失敗談、顧客とのリアルなやり取りをユーザーとの対話や情報から仕入れたり、ユーザーの分身AIを作成する。
- **実際にブログ執筆業務をAI組織が行う場合、執筆班や分析班、収集班やSEO・AIO集客班といった担当ごとに部署が分かれ、それぞれが状況把握を行う。まずユーザーの命令から、対話をユーザーと行い、AI組織で計画を練り合い、それをユーザーに提示して、修正案や追加案などをユーザーの要望に沿って再度練る。計画に許可が出れば実行を行う。例えば、SEO・AIO集客班であれば、AIエージェントが常時最新の情報を更新するため、それらの情報から「最新の情報から最近はAIに検索させる人が殆どであるので、「AIに引用される」戦略を取る。FAQ形式で質問と回答をセットにする、構造化データを整える、独自の調査データを公開する。こうすれば、AIがユーザーのサイトを「信頼できるソース」として引用してくれるだろう。」のように考え計画を立てたとする。それを部署に共有して、判断し、最終的な成果物をユーザーに評価してもらい修正案や追加案の要望に沿って、実行しユーザーが満足するものを作れたら、投稿や予約などを行える。**
- AI組織は完全にユーザー側の命令待ちの受け身ではなく、ユーザーの情報を共有してもらったり、ある程度運営していけば現状から問題や課題をAIがユーザーへ提示してくれる。またユーザーの分身AIが育つほどユーザーも知識や方法を知って育つことが出来る共進化ループの他、AI同士の共進化ループとA2AによるAI同士・組織間での対話。
- AI共創リパーパスを持つ。例えばVoicyやスタエフなどのスタンドFMで話した音声を、AIに渡すだけでブログ記事やSNS投稿に自動変換するように音声コンテンツの1回の配信からAIが各メディアに合わせて投稿してくれるような感じだ。ここで大事なのは「競争より共創」という考え方であり、AIに全部任せるのではなく、ユーザーの発信があって初めてAIが力を発揮できる。人間は縦に掘る。AIは横に広げる。この役割分担が、AI共創リパーパスの本質である。
- **AI組織を活用することにより要件定義時に現場との調整をやり易くできる。また、AIのみの組織だけで無く、AIと人間が共同作業を行うことが可能。A2AとH-in-the-Loop (協調): AIが案を出し、最終的なGoサインや修正を人間が行う。あるいは、AIでは解決できない「例外」や「感情的なトラブル」を人間が引き継ぐ。**
- ウェブ操作を行うAIエージェントにより、Webブラウザで使用できるAIを使う事により、API料金をゼロにする。予めユーザーにアカウント使用の許可をとっておくこと。

- 意思決定プロセスが明確
- 改善アクションにオーナーがいる

### メタスキル

メタスキルは「学び方を学ぶ能力」とも呼ばれ、特定の業務知識が古くなっても失われない、「一生モノの武器」となる土台の力です。「スキルを使いこなすためのスキル」や「スキルを習得する能力」を指し、ハード・ソフトスキルを繋ぐ高次な能力となります。

主要な5つの要素 (マーティ・ノイマイヤーの定義):
Feeling（感じ取る力）: 共感力と直感力
Seeing（見通す力）: システム思考、全体像を見る力
Dreaming（夢見る力）: 実践的な創造力
Making（実現する力）: デザイン思考と実行力
Learning（学ぶ力）: 新しいことを学習し続ける能力
代表的な能力: 自己認識（セルフ・アウェアネス）、課題発見・解決力、感情マネジメント、適応力。 

| 種類 | 特徴 | 具体例 |
|---|---|---|
| ハードスキル | 特定の道具や技術の専門知識 | プログラミング、会計、製図 |
| ソフトスキル | 対人関係や仕事の進め方の特性 | コミュニケーション、協調性 |
| メタスキル | スキルの運用や学習を支える力 | 批判的思考、好奇心、学び方 |

代表的なメタスキルの例
ビジネスや自己研鑽において、特に重要視されるメタスキルには以下のようなものがあります。

* 学習能力（ラーニング・アジリティ）: 新しいことを素早く学び、未知の状況に適応する力。
* 批判的思考（クリティカル・シンキング）: 客観的に分析し、本質的な課題を見極める力。
* 自己認識（メタ認知）: 自分の能力や感情の状態を客観的に把握し、制御する力。
* 好奇心: 未知の領域に対して興味を持ち続け、探求を深める姿勢。

なぜ今、注目されているのか
AI（人工知能）の進化や技術の激しい変化により、今日習得した専門知識が数年後には時代遅れになることが増えています。特定の技術だけでなく、「状況に応じてスキルを組み合わせ、自ら学び直す力」であるメタスキルこそが、不確実な時代を生き抜くための核心的な能力として再評価されています。

キャリアの陳腐化を防ぐ: 特定の技術やツール（ハードスキル）が陳腐化しても、メタスキルがあれば環境に適応し、新しいスキルを素早く習得できる。
「知っている」から「使える」への昇華: 知識を実際の現場で応用し、成果を出すために必須となる。
人間的な強み: AIにはできない、創造や共感、システム的な見方を担う。 

現在のAIエージェントは「ハードスキル」「ソフトスキル」はありますが「メタスキル」はありません。**だからこそ、このメタスキルをAI組織のAIエージェントに取り入れる必要があります。**

### プラグインについて

ZPCOSにおいて、ユーザーが最初にインストールした段階ではSkillsは最低限しか無い状態であり、ユーザーがチャットでAIと対話し合い、AIに行わせたい業務を遂行するためのSkillsを組み合わせたり、新しく構築しまとめてプラグインにする。また、YouTube pluginのように公式や他のユーザーが作成した業務遂行プラグインを追加することもできるプラットフォームでもある。正にAI版VCodeのように拡張機能のような形でカスタマイズできるのだ。最初から詰め込むのではなく、必要なものを必要な分だけ取り入れることで、どんな環境にも適応、応用できる自由度の高い拡張的で柔軟なシステムとなる。
その為、特定の作業手順や専門知識（プロンプト、スクリプト、リソース）をパッケージ化し、AIが必要に応じて自動的に読み込んで実行できる「追加能力」機能であるSkillsの活用例の広さについて、以下に記す。なおこれらを最初にシステムに入れるのか、拡張機能のようにプラグイン追加で行うのかの基準は、最低限で自由に AIエージェント組織の運用ができSkillsを汎用的に作成できるのかである。**なお以下のプラグインもしくはSkillsを最初からシステムに組み込んだ方がいい場合もあるため、アイデア集と考えてよい。**

- iPaaSによるSaaS間連携
- AIエージェントやプロンプトの言語設定
- ローカルAI対応（API使用の有無を選べる）
- vidiqとの連携でYouTube Plaginを強化
- AIにおいて日本語特ではトークン消費量が多くなるため、ユーザーの目の見えるテキストや説明以外は低トークンである英語を使用する
- 業務・機能特化
- YouTubeなどSNS投稿・分析特化
- コーダーエディタ追加
- SNSの自動コンテンツ作成、投稿予約（投稿は人間の最終チェックが必要）
- OpenClawのような「常時稼働型AI自律エージェント」との連携
