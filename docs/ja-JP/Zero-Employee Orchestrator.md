## 1. 用語の定義

### 1.1 Skill / Plugin / Extension の違い

Zero-Employee Orchestrator では、機能追加の単位を次の3層で整理する。

#### Skill
AI が特定タスクを実行するための最小単位の能力。  
プロンプト、スクリプト、手順、制約、利用ツールなどを含む。

- 例: ファイル整理、翻訳、台本生成、競合分析
- 役割: 単一タスクの実行能力

#### Plugin
複数の Skills や補助ツールをまとめて配布・管理するパッケージ。  
特定業務を成立させるための機能セットであり、VS Code の拡張機能に近い感覚で扱う。

- 例: YouTube Plugin、ブログ運用 Plugin、AI 秘書 Plugin
- 役割: 特定業務を実現するためのまとまり

#### Extension
アプリ本体の動作環境、UI、接続先、開発体験を拡張する仕組み。

- 例: Cowork 風 UI、MCP 接続、OAuth 対応、通知機能、クラウド連携
- 役割: ソフト全体の利便性・接続性・開発体験の向上

### 1.2 まとめ表

| 項目 | Skill | Plugin | Extension |
|---|---|---|---|
| 主な対象 | AI の実行能力 | 業務機能のまとまり | アプリ全体の拡張 |
| 単位 | 最小 | 中間 | 広範 |
| 内容 | 単一タスクの手順・知識 | Skills と補助ツールの詰め合わせ | UI、接続、設定、開発体験の強化 |
| 例 | 翻訳、分析、整理 | YouTube Plugin | Cowork 風 UI、MCP、通知 |

### 1.3 重要な前提

### 1.3.1 配布・導入経路の前提

- Plugin / Extension の導入元は、ローカルファイル、Git リポジトリ、将来的な Registry / Marketplace など複数経路を想定する。
- 導入時には、配布元、バージョン、依存関係、必要権限、外部接続先、安全性検査結果を一体で確認できるようにする。
- Webhook、外部イベント、スケジュール起動などの Trigger ベース実行を Plugin / Extension で扱える構造を持つ。
- Skill / Plugin / Extension の配布形式は、本体更新と切り分けて独立に更新可能であることを重視する。
- 非エンジニアでも導入可否を判断できるように、一文要約、用途、危険操作、承認必須操作を導入前に表示する。


- Skills を単体で追加するだけでなく、既存 Skills や新規 Skills を組み合わせて Plugin として追加できるようにする。
- Plugin は VS Code の拡張機能のように、インストール・削除・共有・公開できるようにする。
- Zero-Employee Orchestrator は、チャットによる自然言語だけで Skill / Plugin を設計・生成・修正できることを目標とする。
- 共有・公開時には個人情報、機密情報、認証情報を含めない設計が必要である。
- ローカル保存、Git 管理、クラウド保存は用途に応じて選択可能にする。
- AI による Git 管理、ローカル保存、GitHub や Google Cloud などのクラウド保存を運用方針に応じて切り替えられるようにする。
- 共有・公開される Skills / Plugins は最新版取得や更新確認ができるようにする。

## 2. プロジェクト要約

### 2.1 一文定義

Zero-Employee Orchestrator は、自然言語で業務用 Skill を設計し、複数 AI を組織的に協働させ、人間の承認と監査可能性を前提に業務実行を支援する AI オーケストレーション基盤である。

### 2.2 結論

Zero-Employee Orchestrator は、自然言語による Skill 設計とマルチ AI 協調実行に対応した、軽量かつ拡張性の高い汎用業務オーケストレーションシステムである。  
エンジニア向けには CLI / TUI、非エンジニア向けには GUI を提供し、どちらも共通の AI 組織基盤の上で動作する。

### 2.3 目的

Zero-Employee Orchestrator の目的は、マルチ AI エージェントと人間が共存・協調しながら、**計画、実行、検証、改善、監査、再利用**を一つの環境で回せる AI 組織基盤を実現することである。

特に重視するのは次の5点である。

- AI を単体ツールではなく「組織」として運用できること
- AI の判断や行動をブラックボックス化させず可視化すること
- 人間の最終承認を残しつつ、業務の大部分を自律実行できること
- 権限、監査、保守運用を含めて企業利用に耐えること
- 新しい AI サービス、API、モデル変化に長期的に追従できること
- 既存 AI エージェントが抱える過剰権限、監査不透明、継続稼働リスクに企業が対応できること

### 2.4 代表ユースケース

最も先鋭的なユースケースは、経営者 1 名が指示と承認のみを行い、実務は AI 組織が担う**ゼロパーソンカンパニー**の運営である。  
ただし Zero-Employee Orchestrator はその専用ソフトではなく、以下のような幅広い用途に対応する汎用基盤として設計する。

- 会社業務全般の自律化
- AI 研究・実験
- コーディング支援
- マーケティング・SNS 運用
- クリエイティブ制作
- バックオフィス補助
- 一般ユーザー向け AI 自動化
- 将来的な AI ロボット、スマート機器、遠隔操作環境

### 2.5 ゼロパーソンカンパニーとの関係

ここで言うゼロパーソンカンパニーとは、**社長という意志決定者は存在するが、組織として多数の人員を雇わずに会社を回せる状態**を指す。  
Zero-Employee Orchestrator は、その状態を成立させるために必要な**計画・実行・監査・承認・改善**の仕組みを、一つのソフトウェア環境に統合することを目指す。

### 2.6 Zero-Employee Orchestrator が対象とする利用者層

Zero-Employee Orchestrator は、単なる開発者向けツールでも、単なる一般ユーザー向け自動化ツールでもない。  
主な対象は次の3層である。

- **非エンジニアの実務者・経営者**  
  自然言語で業務を依頼し、AI 組織の計画・実行・承認を管理したい利用者
- **エンジニア・研究開発者**  
  Skills / Plugins / Extensions を開発し、業務自動化基盤を拡張したい利用者
- **将来的な組織利用者**  
  監査、権限管理、承認、履歴保存を前提に AI を業務に導入したい組織

そのため Zero-Employee Orchestrator は、**GUI だけでも使えること**と、**CLI / TUI でも深く制御できること**の両立を目指す。

## 3. Zero-Employee Orchestrator の中核思想

### 3.1 AI を「道具」ではなく「組織」として扱う

Zero-Employee Orchestrator は、単一の AI チャットや単発自動化ではなく、複数の AI が役割分担し、相互監視し、情報共有しながら動く AI 組織を構築することを目指す。

### 3.2 人間の最終承認を外さない

重要操作、外部送信、公開、投稿、課金、権限変更、プライバシーに関わる処理については、必ず人間の確認と承認を必要とする。

### 3.3 ブラックボックスを減らす

以下を常にユーザーが確認できるようにする。

- どの AI が何を担当しているか
- 何を根拠に計画したか
- どこまで完了したか
- どのモデルや API を使ったか
- なぜ差し戻されたか
- なぜ再計画が発生したか
- どの Skill / Plugin / Extension が使われたか

### 3.4 最新技術は本体ではなく拡張で取り込む

アプリ本体の更新頻度は抑えつつ、最新技術や新しい業務対応は Skills / Plugins / Extensions 側で追加できる構造を重視する。

### 3.5 安全性と信頼性を最優先する

Zero-Employee Orchestrator は、AI を無制限に自律化するためのソフトではなく、権限、監査、ルール、承認を備えた**高信頼な AI 組織運営基盤**を目指す。

### 3.6 OS と呼ぶ理由

Zero-Employee Orchestrator は、単体タスクの自動化だけでなく、業務同士の接続、成果物の再利用、履歴管理、権限管理、再計画、監査、拡張導入までを一つのアプリケーション上で扱う。  
そのため、単発の AI エージェントではなく、**業務全体を支える OS 的基盤**として位置づける。

### 3.7 Agent Teams と Subagents の考え方

Zero-Employee Orchestrator では、複数 AI の協働を単純な「サブエージェント呼び出し」で終わらせず、**役割分担されたチーム運用**として扱うことを重視する。

#### Subagents
- 主エージェントから指示を受けて部分作業を行う
- 一方向の報告関係になりやすい
- 単純な補助、委譲、短い処理に向く

#### Agent Teams
- 複数 AI が役割ごとに担当を持つ
- 相互に情報共有、確認、批評、再提案を行う
- 複雑な業務、長い工程、複数成果物、継続運用に向く

Zero-Employee Orchestrator は、単なる subagent 呼び出しではなく、**計画・実行・監督・検証を分担した Agent Teams 的な構造**を重視する。  
また将来的には、ユーザーが**チーム単位だけでなく個別メンバー単位でも介入・差し戻し・指示変更できる**ことを目指す。

## 4. 利用時の基本フロー

### 4.0.1 spec / plan / tasks 中間成果物の保存単位

Zero-Employee Orchestrator では業務実行の透明性と再利用性を高めるため、
すべての実行を **spec / plan / tasks** の3段階成果物として保存する。

保存は案件単位または業務単位で行われ、以下の構造を基本とする。

project/
 ├ spec/
 ├ plan/
 ├ tasks/
 ├ outputs/
 ├ review/
 └ logs/

この構造はソフトウェア開発専用ではなく、
調査業務、SNS運用、資料作成、分析業務など
あらゆる業務で共通して利用できる。



1. ユーザーが自然言語で目的や業務を入力する。  
2. AI 組織が要件を深掘りし、実行計画を作成する。  
3. 計画には工程、担当 AI、想定時間、推定 API コスト、必要権限、外部接続の有無、危険操作の有無を含める。  
4. ユーザーが計画を確認し、修正・追加・削除を指示する。  
5. 権限チェック、危険操作警告、コスト確認を経たうえで承認された場合のみ実行に進む。  
6. 実行中は進捗、成果物、失敗、自己修復履歴、利用中の Skill / Plugin / Extension を可視化する。  
7. 完了後、成果物をユーザーが確認し、差し戻しまたは承認する。  
8. 会話、判断、成果物、改善履歴を記録し、次回以降の運用に活かす。  

### 4.0.2 spec / plan / tasks 形式での中間成果物管理

Zero-Employee Orchestrator では、実行前の計画を単なる会話ログとして終わらせず、必要に応じて中間成果物として構造化保存する。  
参考にできる実践として、次の3段階を採用する。

- **spec**: 何を達成するか、目的、制約、受け入れ条件
- **plan**: どう実現するか、工程、利用する AI / Skill / Plugin / 外部接続、承認ポイント
- **tasks**: 実行可能な作業単位、担当、依存関係、検証条件、完了条件

ただし Zero-Employee Orchestrator では、これを**ソフトウェア開発専用の設計書形式ではなく、汎用業務にも使える計画記述形式**として扱う。  
たとえば、SNS 運用、競合調査、資料作成、バックオフィス処理でも、目的・制約・工程・担当・検証条件・人間承認ポイントを `spec / plan / tasks` に準じた形で記録できるようにする。

これにより、会話依存のブラックボックス性を減らし、再利用性、監査性、差し戻しや再実行のしやすさを高める。

### 4.1 計画時に明示すべき追加情報

### 4.1.1 イベント駆動実行で明示すべき情報

Zero-Employee Orchestrator では、ユーザーが自然言語で開始する流れに加えて、Webhook、外部イベント、定期実行などのイベント駆動実行にも対応可能な構造を持つことが望ましい。  
その場合でも、無条件に自律実行させるのではなく、次の情報を計画または設定段階で明示する。

- 起動条件
- 起動元
- 自動実行される範囲
- 人間承認が必要な工程
- 失敗時の停止条件
- 再試行条件
- 外部送信の有無
- 使用する認証方式

これにより、イベント駆動であってもブラックボックス化を防ぎ、承認と監査の整合性を保つ。


Zero-Employee Orchestrator の計画提示では、工程、担当 AI、費用、権限、危険操作だけでなく、必要に応じて次の情報も明示する。

- 使用予定モデルの候補と選定理由
- 高品質重視 / 速度重視 / 低コスト重視 / 無料枠優先などの実行モード
- 失敗時の代替ルート
- 外部参照元の種類
- ローカルのみで完結する工程と外部接続が必要な工程の区別
- 実行後に人間承認が必要な箇所の一覧

これにより、ユーザーは「何が起きるか分からないまま実行する」状態を避けられる。

### 4.2 自然言語トリガーと内部コマンドモデル

Zero-Employee Orchestrator では、自然言語だけで基本操作を進められることを重視する。  
その一方で、繰り返し使う操作については、自然言語と内部コマンド体系を対応づけることで一貫性を高める。

たとえば内部的には、次のような操作概念を持てる。

- 依頼内容から Interview を始める
- 計画化を始める
- 承認待ちに移す
- 実行を開始する
- 次のタスクだけ進める
- レビューを要求する
- 完了処理に進める
- 履歴や進捗を同期する

ただし Zero-Employee Orchestrator では、これを**CLI 専用の開発者操作体系に閉じず、GUI と自然言語の両方から同じ意味で扱える操作モデル**として実装する。


自然言語入力と内部操作の対応例
- 「この業務を整理して」→ Interview 開始
- 「計画を出して」→ spec / plan の生成
- 「承認待ちにして」→ review 待ちへ遷移
- 「次だけ進めて」→ 単一 task 実行
- 「見直して」→ Judge / Re-Propose 実行
- 「終わらせて保存して」→ 完了処理・履歴保存

## 5. 使用時の想定挙動

- ユーザーは AI と対話しながら、業務の目的、条件、制約、優先順位を詰める。
- AI 組織は内部情報と外部情報を参照しながら、実行計画を立案する。
- 計画時には、どの工程でどの AI や API を使うか、どの程度の費用がかかるか、どの権限が必要かを明示する。
- ユーザーはその計画に対して批評や修正を加え、納得した場合のみ実行を許可する。
- 実行時には役割や工程ごとに AI が担当班として割り振られ、進行状況や課題を相互に監視する。
- その過程をブラックボックス化せず、思考の要約、判断根拠、実行ログ、差し戻し理由をユーザーが確認できるようにする。
- 実行中の追加指示や修正依頼にも対応できるようにする。
- 納品、投稿、公開、外部送信などの最終操作は必ず人間が承認する。
- 実行履歴、会話履歴、判断履歴、成果物は記録し、将来的な分身 AI や AI 秘書、経験学習に活用できるようにする。
- 計画時には必要に応じて Skills を組み合わせたり新規作成したりして、タスク専用の Plugin を構築できるようにする。
- Skills や Plugins は、個人情報や機密情報を除いたうえで共有・公開可能にする。
- コンテキストウィンドウの制限に対応するため、履歴管理機構と必要情報の再取得機構を設ける。

## 6. MVP 定義

### 6.1 MVP の目的

MVP の目的は、Zero-Employee Orchestrator の本質である**自然言語による業務設計、複数 AI による計画・実行・検証、人間承認、可視化された運用**を、最小構成で end-to-end に成立させることである。

### 6.2 MVP に必須の機能

- 自然言語入力による業務依頼
- Design Interview による要件深掘り
- Plan / DAG の生成
- API コストの概算提示
- 必要権限と外部接続の提示
- Skill 割当と基本的な Skill 実行
- Judge による品質確認
- 差し戻しと再提案
- 人間承認フロー
- 実行ログ・成果物・履歴の保存
- GUI での確認画面
- 最低限の認証情報保護
- Local Context の安全な読み込み
- 実証用 Plugin を 1 系統以上搭載
- 本体機能と Plugin / Extension の境界が最低限整理されていること

### 6.3 MVP で対象外とするもの

- 大規模な公開 Registry 経済圏の完成
- 完全自律での無承認外部実行
- 大量の企業向け権限階層の完全実装
- すべての業務領域への対応
- 高度な分身 AI の完成
- 大規模なマルチテナント運用
- 高度なロボット制御や物理機器連携
- 外部 SaaS の認証情報自動再発行

### 6.4 MVP の成功条件

- ユーザーが自然言語だけで業務依頼を開始できる
- AI 組織が実行計画を提示し、ユーザーが修正・承認できる
- 実行から成果物生成まで一連の流れが通る
- Judge によって最低限の品質確認が行われる
- 外部送信や重要操作が人間承認なしに進まない
- 進捗と実行内容がユーザーに見える
- 失敗時に再提案または安全停止できる

### 6.5 MVP における代表デモ条件

MVP は、概念説明ではなく実動デモとして成立している必要がある。  
少なくとも次の条件を満たすことを目標とする。

- 自然言語入力から Design Interview が開始する
- AI 組織が計画と担当分担を提示する
- コスト、権限、危険操作の有無が表示される
- 実行中に進捗と成果物が見える
- Judge による品質判定が行われる
- 差し戻し時に再提案または再実行が起こる
- 最後の公開、送信、投稿は人間承認を要求する
- 実行後に履歴、成果物、判断情報が保存される

## 7. 権限モデル

Zero-Employee Orchestrator では最低限、以下のロールを想定する。

### 7.1 Owner
- 全権限を持つ
- システム設定、認証情報管理、Plugin 導入、公開設定、承認、監査ログ閲覧が可能

### 7.2 Admin
- 組織設定、Plugin 導入、一部承認、監査ログ閲覧が可能
- 認証情報の一部編集権限を持つ場合がある

### 7.3 User
- 業務依頼、計画確認、成果物確認、差し戻しが可能
- 危険権限の付与や外部公開はできない

### 7.4 Auditor
- 実行履歴、承認履歴、監査ログの閲覧が可能
- 実行や編集権限は持たない

### 7.5 Developer
- Skill / Plugin / Extension の作成や修正が可能
- 本番導入や公開には別途承認を必要とする

## 8. データ保存・メモリ方針

### 8.0.1 改善知識の分離保存

会話ログそのものと、再利用可能な知識は分離して保存する。

保存は次の4種類に分類される。

- conversation log（会話履歴）
- reusable improvement（再利用可能な改善知識）
- experimental knowledge（実験的知識）
- verified knowledge（検証済み知識）

再利用可能な知識は、タスク種別、Plugin種別、ドメイン種別ごとに整理され、
個人依存の文脈は可能な限り除去する。



### 8.1 保存対象

- 会話履歴
- 実行計画
- 実行ログ
- 承認履歴
- 成果物
- エラー履歴
- 利用した Skill / Plugin / Extension の情報
- ユーザー設定
- 失敗分類と回復履歴

### 8.2 保存対象外または厳格管理対象

- API キー
- OAuth トークン
- 決済情報
- 個人情報
- 社外秘情報
- 外部共有してはいけない内部文書

これらは暗号化・分離保存し、通常ログには平文で残さない。

### 8.3 メモリ利用方針

- Experience Memory は、成功した計画や差し戻しから学んだ改善パターンを保存する。
- 分身 AI や AI 秘書に転用する記録は、ユーザーが保存範囲を制御できる設計とする。
- メモリには削除、リセット、エクスポート機能を設ける。
- 共有・公開時には個人情報、機密情報、認証情報を除外する。

### 8.4 保存・共有時の情報サニタイズ方針

Zero-Employee Orchestrator では、保存・共有・公開の各段階で、情報をそのまま扱わない。  
少なくとも次の処理方針を取る。

- API キー、OAuth トークン、シークレット値はマスキングまたは参照 ID 化する
- 個人名、住所、決済情報などの個人情報は共有時に除去または匿名化する
- 社外秘資料、未公開情報、契約情報は公開対象から除外する
- 会話ログや改善ログを学習や共有に使う場合は、ユーザーが範囲を制御できるようにする

共有文化を成立させる前提として、**安全なサニタイズ処理**は中核要件である。

---

### 8.5 改善知識の分離保存方針

Zero-Employee Orchestrator では、単なる会話履歴の保存だけでなく、**改善知識を用途ごとに分離して保存する**ことを重視する。  
たとえば次のように整理できる。

- **Experience Memory**: 成功した実行パターン、再利用可能な判断
- **Failure Taxonomy**: 失敗分類、よくある差し戻し理由、再発防止策
- **Policy Memory**: 承認条件、禁止事項、権限ルール
- **Skill Improvement Memory**: 特定 Skill の改善点、実務上の癖、品質基準
- **Plugin Operation Memory**: 業務単位での運用ノウハウ、必要な確認順序

この設計により、単に「前回の会話を思い出す」だけではなく、**改善がどの層に属するかを明示したうえで再利用**できるようにする。


また、改善知識を再利用する際には、**誰が承認した知識か、どの実行条件で有効だったか**を紐づけて保存する。  
これにより、偶然うまくいった経験則と、再利用に耐える運用知識を区別しやすくする。

- インストーラーによる導入

- 将来的な自動アップデート機構
- 更新前のバックアップやロールバック可能性
- 本体更新と Plugin / Extension 更新の切り分け

特に本体は安定性を重視し、業務対応や新機能追加は Plugin / Extension 側で吸収できる構造を優先する。

### 9.5 可観測性の責任分界

Zero-Employee Orchestrator では、ローカル / クラウド / 外部サービスの責任分界に加え、**可観測性の責任分界**も明確にする。

## 11. 実装・運用で追加考慮すべき技術要件

### 11.0.1 既存業務への段階導入（Brownfield対応）

Zero-Employee Orchestrator は白紙環境のみを前提としない。
既存業務・既存ツール・既存データを前提とした導入が可能である。

導入は次の段階で行う。

1. 観察（既存業務の可視化）
2. 補助（AIによる補助作業）
3. 半自動化
4. 自律実行

既存業務を破壊せず、段階的に置き換えることを基本方針とする。



以下は設計上の重要論点である。

### 11.1 モデル接続と切替

- OpenRouter に閉じず、複数 Provider を直接または AI Gateway / Proxy 経由で接続できるようにする。
- Bedrock、Azure OpenAI、Vertex AI、Cerebras、Cloudflare AI、Z.ai、Ollama、LM Studio などへの接続可能性を考慮する。
- Claude Code や Codex のような公式 CLI / 公式接続は、利用規約に沿う範囲で統合利用できるようにする。
- ローカルモデル、API モデル、サブスク系公式ツールを併用し、コスト・速度・品質・規約適合性のバランスを取る。

- OpenRouter だけに固定せず、複数の LLM Provider、AI Gateway、ローカル LLM に対応可能な構造を維持する。
- LiteLLM Gateway などを用いて、OpenAI 互換形式で複数モデルを切替可能にする。
- 高品質重視、速度重視、無料枠重視、低コスト重視などのモード選択を可能にする。
- モデル変更時にアプリ側の大規模改修を必要としない設計を採る。

### 11.2 AI Gateway / コスト管理

- API コスト見積りだけでなく、キャッシュ、利用制限、失敗時の代替提案も考慮する。
- モデル料金、特性、更新情報を把握するための共通データ置き場を持つ。
- 情報更新そのものが API コストを増やしすぎないようにする。
- AI 自体が無駄な API 呼び出しを減らすよう改善できる構造を持つ。

### 11.3 プロキシ・企業ネットワーク対応

- 社内プロキシや VPN を経由しないと接続できない環境を想定する。
- 証明書エラーや接続制限に対応するための設定層を用意する。
- これらは本体に過剰に埋め込むのではなく、Extension で補強可能にする。

## 12. Plugin / Skill 審査方針

### 12.0.1 Verified Skill / Plugin 判定基準

Skill / Plugin は次の4段階で状態管理する。

- experimental
- private
- verified
- deprecated

verified と判定されるためには次を満たす必要がある。

- 安全性検証
- 権限使用範囲の明示
- 外部接続先の明示
- 再現可能なテスト結果
- 監査ログの確認



### 12.1 基本方針

- 攻撃的、破壊的、有害な Plugin / Skill は作成・導入・共有できない。
- 新規 Skill / Plugin は導入前に静的検査と権限確認を行う。
- 外部通信先、必要権限、ファイル操作権限、外部送信有無を明示する。

### 12.2 検査項目

- 危険なコードの有無
- 危険な関数やコマンドの使用
- 外部送信の有無
- 認証情報アクセスの有無
- 削除、更新、公開操作の有無
- 実行対象の範囲
- 使用ライブラリの妥当性
- 同意画面フィッシングや危険な設定誘導の有無
- 秘密情報漏洩の可能性


### 12.2.1 Skill / Plugin 間の指示競合解決

複数の Skill / Plugin を同時に使う場合、**同じ対象に対して矛盾した行動指示**が発生しうる。  
Zero-Employee Orchestrator では、これを単なる「相性問題」として放置せず、**検出・分類・解決**の対象として扱う。

競合の典型例
- ある Skill は「高速実行を優先」と指示し、別の Skill は「品質重視で多段検証を必須」と指示する
- ある Plugin は特定ツールチェーンを要求し、別の Plugin は別方式を前提とする
- ある運用 Skill は投稿を即時実行しようとし、別の Policy Skill は承認必須としている

競合が見つかった場合の優先順位は、原則として次の通りとする。
1. **人間が明示した指示**
2. **権限・安全・法務・組織ポリシー**
3. **案件固有の spec / plan / tasks**
4. **verified な Skill / Plugin の明示ルール**
5. **experimental / private な拡張の補助指示**

さらに、各 Skill / Plugin には可能な限り次の情報を持たせる。
- 適用対象
- 前提条件
- 非互換条件
- 優先度
- 上書き禁止ルール
- 承認必須条件

これにより、単純な多数決ではなく、**監査可能な競合解決**を実現する。

### 12.3 強制承認対象

以下を含む場合は、人間承認を必須とする。

- 外部送信
- 投稿
- 公開
- 削除
- 課金
- Git push / release
- 重要ファイルの上書き
- 権限変更
- API キー関連処理
- 外部認証情報の更新反映

### 12.4 Skill / Plugin 導入前の実務チェック

- その機能は本体に入れるべき最小核か、Plugin / Extension に切り出すべき拡張か。
- 自然言語から再生成・再構築できる領域か。
- ユーザーが工程追加・削除・API 差し替えを自然言語で指示できるか。
- 共有配布しても安全か、利用規約違反を誘発しないか。

導入前の安全検査では、コード上の危険性だけでなく、実務上の事故も防ぐ必要がある。  
そのため、次の観点も確認対象とする。

- ユーザーに誤認を与える説明になっていないか
- 不要に強い権限を要求していないか
- 本番環境と開発環境を混同していないか
- 外部送信先や保存先が明示されているか
- 更新時に既存データや設定を壊さないか
- 認証情報ローテーション後も正常に動くか


### 12.5 Skill / Plugin に必須の透明性情報

Skill / Plugin は、インストールできること自体ではなく、**何をするかが説明可能であること**を最低条件とする。  
そのため、各 Skill / Plugin には少なくとも次の情報を持たせる。

- 一文要約（何をする Skill か）
- 想定ユースケース
- 必要権限
- 外部接続先
- 読み取り対象
- 書き込み対象
- 危険操作の有無
- 承認必須操作
- 非互換な他 Skill / Plugin
- 想定コスト帯
- バージョン
- 発行者または管理主体

この情報が不十分な Skill / Plugin は、たとえ便利でも verified とみなさない。

### 12.6 Skill / Plugin の由来・版管理・承認記録

Zero-Employee Orchestrator では、導入済み Skill / Plugin について**由来と承認経路を追跡できること**を重視する。

最低限保持すべき情報
- 導入元
- 取得日時
- バージョンまたはコミット識別子
- 承認者
- 承認日時
- 使用可能な権限範囲
- 最終監査日時
- 失効または停止理由

共有 Registry を使う場合でも、利用側では**バージョン固定、承認履歴、更新差分の確認**を行う。  
これにより、後から「誰が、いつ、どの版を、どの権限で許可したか」を説明できる状態を保つ。

## 13. セキュリティ方針

### 13.1 セントラルリポジトリの考え方

モデル更新情報、料金、特性、推奨用途、注意事項などを毎回個別取得すると、コストや更新負荷が大きくなる。  
そのため Zero-Employee Orchestrator では、こうした情報を集約する**セントラルリポジトリ**の考え方を持つ。

このリポジトリは、例えば次の情報を保持する。

- モデル料金
- モデル特性
- 利用可能 Provider
- 推奨用途
- 制約事項
- 更新日
- 既知の不具合や注意点

これにより、各 Skill や Plugin が個別に同じ情報を取得し続けることを避け、API コストと更新負荷を抑える。

### 13.2 参考にする外部ツール・接続先の例

Zero-Employee Orchestrator は独自実装に閉じるのではなく、既存の有用なツールや接続方式を必要に応じて取り込む。

参考対象の例:
- LiteLLM
- OpenRouter
- Ollama
- MCP
- Helicone
- Portkey
- n8n
- Claude Code
- Codex
- GitHub Copilot

これらはそのまま採用する対象ではなく、**どの思想・機能・接続性を取り込むか**を見極めたうえで活用する。

### 13.3 セキュリティ方針

- API キーや認証情報は暗号化して管理する。
- 通常ログに平文の機密情報を出力しない。
- 悪意あるウェブサイトや危険な外部リソースへの接続を制限する。
- 使用中の AI サービス、ツール、API の仕様変更、利用規約変更、セキュリティ情報を公式ソースから取得する仕組みを持つ。
- RSS 等による公式情報の定期取得を行えるようにする。
- API キーなどの重要情報を定期的に更新しやすいよう、更新支援用の Skill / Plugin / Extension を用意する。
- 自組織内の安全確認のための防御的セキュリティテストは許可するが、攻撃的用途や第三者対象の挙動は許可しない。
- 追加導入される Skill / Plugin / Extension は、事前に安全性検証を行う。
- Skill のマージ前に静的解析、依存関係確認、秘密情報漏洩検査を行う。
- 会話履歴、改善ログ、共有物に秘密情報の実値を残さない。


### 13.4 Skill 設定ファイル・ポリシーファイル変更の監査

Skill / Plugin 本体だけでなく、AI の行動規則を変える設定ファイルも**高リスク変更**として扱う。  
たとえば、次に相当するファイルや設定はセキュリティ監査対象とする。

- Skill 定義ファイル
- Policy Memory
- 承認ルール
- 禁止操作ルール
- 外部送信ルール
- Agent Team の役割定義
- プロジェクト固有の行動指針ファイル

これらの変更は通常の UI テキスト編集と同列に扱わず、
差分表示、承認履歴、変更理由、ロールバック可能性を必須にする。  
Zero-Employee Orchestrator では、**AI の振る舞いを変える変更はコード変更と同等以上に重要**とみなす。

## 14. 認証情報ローテーション方針

Zero-Employee Orchestrator は、外部 AI サービスやクラウド連携に用いる認証情報について、**自動更新**ではなく**安全なローテーション支援**を基本方針とする。

### 14.1 基本原則

- API キー、OAuth、サービスアカウントなど、認証方式の違いを区別して管理する。
- 期限監視、権限棚卸し、更新提案、反映確認を一連のフローとして扱う。
- 料金、権限、公開設定、本番影響がある変更は必ず人間承認を必要とする。
- 秘密情報の実値は表示、保存、共有を最小化し、原則として Secret Manager や OS 標準の安全な保管先に置く。

### 14.2 想定フロー

1. 接続先ごとに認証方式、権限範囲、作成日、最終確認日、適用環境を台帳化する。  
2. 一定期間ごとに、長期未更新、過剰権限、開発・本番共用などのリスクを検出する。  
3. ユーザーに対して、更新推奨理由、影響範囲、差し替え箇所、ロールバック手順を提示する。  
4. 人間承認後に、新しい認証情報の発行手順や差し替え手順を案内する。  
5. 更新後は疎通確認テストを行い、問題がなければ旧キーの無効化を促す。  

### 14.3 実装方針

- 外部 SaaS の API キーを一律に自動再発行する機能は、初期段階では持たない。
- 自動化する場合も、自社管理下の内部 API や Secret Manager 連携済みの資格情報など、限定領域から始める。
- 本体では、認証情報の棚卸し、期限監視、更新提案、疎通確認、監査ログを中核機能とし、サービス別詳細は Plugin / Extension で拡張する。

### 14.4 想定 Plugin / Skill 例

- Credential Hygiene Plugin
- Secret Rotation Assistant Skill
- Post-Rotation Verification Skill

### 16.5 Paperclip との違いと参考にできる点

Paperclip は、zero-human companies のためのオープンソース orchestration を掲げ、org chart、budgets、governance、goal alignment、ticket system、heartbeats、multi-company などを中核にした AI 組織運営基盤として整理されている。  
Zero-Employee Orchestrator も複数 AI を組織的に動かす点では近いが、同一方向の模倣ではなく、より **業務 OS / 監査可能な業務基盤** として再解釈して差別化する必要がある。

#### 16.5.1 主な違い

- Paperclip は「AI 会社を運営するための orchestration」に強く寄っているのに対し、Zero-Employee Orchestrator は、会社運営に限らず、調査、分析、資料作成、SNS 運用、バックオフィスなどを横断して扱う **汎用業務 OS** を目指す。
- Paperclip は goal、org chart、ticket、heartbeat、budget を中心に AI 組織を運営する設計だが、Zero-Employee Orchestrator はそれに加えて、**spec / plan / tasks を中間成果物として保存し、再利用・監査・差し戻し・再実行をしやすくすること**を重視する。
- Paperclip は AI エージェントを会社組織として扱う UI / 管理系に強みがある一方、Zero-Employee Orchestrator は **計画・実行・検証・改善・監査・再利用** を一つの基盤でつなぐことを重視する。
- Paperclip では governance により承認や停止、上書きが可能だが、Zero-Employee Orchestrator ではさらに、**外部送信、公開、投稿、課金、権限変更、認証情報変更などを強制承認対象として明示する安全設計**を重視する。
- Paperclip は BYO Agent 的に外部エージェントを組織へ接続する発想が強いのに対し、Zero-Employee Orchestrator は **Skill / Plugin / Extension を階層的に管理し、本体・業務機能・接続拡張を分離する構造**をより明確にする。
- Paperclip はチケットと監査ログによる可視化に強いが、Zero-Employee Orchestrator はさらに **Judge Layer / Cross-Model Judge / 差し戻しループ** を明示し、AI 生成物の品質保証と検証プロセスを独立した層として扱う。
- Paperclip は Web ベースの運用ダッシュボード色が強いが、Zero-Employee Orchestrator は **GUI / CLI / TUI を共通基盤で扱い、ローカル中心でも運用可能な業務 OS** として設計する。

#### 16.5.2 参考にできる点

Paperclip から参考にできるのは、思想全体のコピーではなく、次のような見せ方・運用原則である。

- **org chart の可視化**  
  AI を単なる並列タスク実行器ではなく、役割・上下関係・責任範囲を持つ組織として見せる UI は参考になる。  
  Zero-Employee Orchestrator でも、Agent Teams、責任分界、承認権限、差し戻し先、監督担当を可視化する設計は有効である。

- **budget / cost control の明確化**  
  Agent 単位・業務単位で予算上限を明示し、超過時に停止または承認待ちにする設計は実務上の説得力が高い。  
  Zero-Employee Orchestrator でも、タスク単位・Agent 単位・Plugin 単位でコストを見える化し、閾値超過時の停止条件を強化する参考になる。

- **heartbeat 的な定期起動モデル**  
  定期実行、状態確認、再開、巡回業務、自動レビューなどを schedule ベースで回す考え方は有用である。  
  ただし Zero-Employee Orchestrator では、単なる cron 的起動ではなく、承認要否、停止条件、再試行条件、エスカレーション条件まで含めて扱う。

- **goal alignment の明示**  
  会社目標 → プロジェクト目標 → Agent 目標 → Task という文脈の継承を見せる考え方は参考になる。  
  Zero-Employee Orchestrator でも、spec / plan / tasks と上位目的の関係を UI 上で見えるようにし、各タスクが「なぜ存在するのか」を追跡可能にすることは有効である。

- **ticket / trace ベースの可視化**  
  やり取り、指示、判断、ツール呼び出し、承認の流れを追跡できる設計は参考になる。  
  Zero-Employee Orchestrator ではこれを ticket に限定せず、会話、計画、実行、レビュー、監査ログを横断して関連づける方向で発展させる。

- **multi-company / control plane の発想**  
  一つの基盤で複数事業、複数組織、複数ブランドを分離運用できる発想は、将来的な拡張先として参考になる。  
  Zero-Employee Orchestrator でも、複数案件・複数事業・複数法人を一つの操作基盤から安全に扱える構造は重要である。

- **Bring Your Own Agent の訴求**  
  既存の外部エージェントやツールを組み込めることを分かりやすく示す見せ方は参考になる。  
  Zero-Employee Orchestrator では、BYO Agent に加えて、BYO Model、BYO Tool、BYO Memory、BYO Execution として整理することで、より本質的な拡張性を示せる。

#### 16.5.3 Zero-Employee Orchestrator 側でさらに強く明文化すべき点

Paperclip のような近い方向性のソフトウェアと比較されたときに、Zero-Employee Orchestrator の独自性を明確にするため、次の点はより前面に出すべきである。

- Zero-Employee Orchestrator は「AI 会社管理ツール」ではなく、**監査可能な業務 OS** であること

- ゼロパーソンカンパニー運営だけでなく、一般業務全般へ適用可能な汎用性を持つこと

#### 16.5.4 取り込み方の原則

Paperclip から参考にする場合も、Zero-Employee Orchestrator は「会社を模した AI 組織の運営 UI」をそのまま目指すのではない。

## 18. Skill Registry と共有方針

### 18.0.1 Registry における検証状態表示

Skill Registry では各 Skill / Plugin に対して
検証状態を明示する。

表示例

- verified
- experimental
- private
- deprecated

これにより利用者は安全性と成熟度を判断できる。

- 共有文化は VS Code 拡張機能に近い体験を目指すが、安全性の確認を優先する。

### 18.1.1 Registry / Marketplace の導入経路表示

Registry では、Skill / Plugin の内容だけでなく、どの経路で導入できるかも分かる状態が望ましい。

- 公式 Registry 導入
- 組織内配布
- 手動インポート

これにより、非エンジニアでも「どこから来た拡張か」を把握しやすくする。

### 18.1.2 導入前サマリー表示

導入前には詳細仕様を読まなくても判断できるように、次の短い要約を表示できることが望ましい。

- 何をするか
- 何に使うか
- どの権限が必要か
- 外部接続があるか
- 人間承認が必要な操作は何か
- 既知の制限は何か


Registry の各項目には、少なくとも次を表示できる状態が望ましい。

- 検証状態
- 対応する業務領域
- 対応する実行環境
- 必要権限
- 外部接続先
- 危険操作の有無
- 非互換な Skill / Plugin
- 最終更新日
- 最終監査日
- 既知の問題
- 依存関係
- 承認必須条件

これにより、利用前の比較検討を会話任せにせず、一覧性を保てる。

### 18.2 Blocklist / Quarantine / 再スキャン

危険と判断された Skill / Plugin は、単に「非推奨」と表示するだけでなく、
次の状態遷移を持てるようにする。

- **warning**: 注意喚起付きで利用可能
- **quarantine**: 新規利用禁止、既存利用も警告
- **blocked**: 実行禁止
- **deprecated**: 代替へ移行推奨

また、初回導入時の検査だけでなく、更新時・依存変更時・ポリシー変更時に**再スキャン**を行う。  
これにより、「導入時は安全だったが、後から危険化した拡張」を見逃しにくくする。

### 25.1 自律実行の境界

Zero-Employee Orchestrator は AI の自律実行を許可するが、
すべての操作を無制限に許可するわけではない。

自律実行可能

- 調査
- 分析
- 下書き作成
- 情報整理

承認必須

- 公開
- 投稿
- 課金
- 削除
- 権限変更
- 外部送信

高リスク操作は必ずユーザー承認を必要とする。

 最低限の安全原則

- 人間承認なしに重大操作を実行しない
- 機密情報を外部共有しない
- 危険な Skill / Plugin / Extension を許可しない
- 監査可能性を失わない
- 履歴削除や改ざんを容易にしない
- ユーザーが停止、確認、差し戻しできる
- AI の誤作動時には安全側に倒す

- 統合前の確認手順
- 衝突時の優先ルール

将来的には、各 AI に対して**担当領域ごとの作業ブランチ、成果物単位の分離、統合前チェック**を持たせられる構造を取り入れる。

## 35. 要望の整理

以下は要望を論点別に整理したものである。

### 35.1 理念・方向性

- システム本体の更新頻度は抑え、最新技術は Skills / Plugins 側で吸収したい。
- AI が AI を指揮するオーケストレーションを実現したい。
- 単機能な AI SaaS ではなく、企業全体の業務をシームレスに扱える基盤にしたい。
- セキュリティ、権限、保守運用まで含めて企業利用に耐える必要がある。
- 企業や個人が、自分たちの現場に特化した AI を容易に作れる環境にしたい。

### 35.2 最重要目的

Zero-Employee Orchestrator の最重要目的は、既存 AI エージェントの最大の課題である「リスク管理と信頼構造の欠如」を解消することである。

具体的には、以下に対応できる基盤を目指す。

- 誰がどの AI に何を任せたかを追跡できること
- 過剰権限を避けること
- 操作ログを明確に残すこと
- 退職・担当変更・運用停止時にも制御できること
- ERP / CRM / セキュリティのような慎重さが必要な領域とも接続できること

- spec / plan / tasks の中間成果物管理  
- 自然言語トリガーと内部コマンドモデルの対応づけ  
- 並列 AI の責務分離と衝突防止  
- 改善知識の階層化保存  
- 状態管理、イベント記録、コスト分析の可視化  
- コーディング系 Plugin 向けの意味的コード理解  

### 35.3 参照したい外部思想

Zero-Employee Orchestrator は、OpenCode の以下の思想を継承対象とする。

- 軽量・高速
- CLI / TUI ベースでも使いやすい
- 低スペック環境でも動きやすい
- 柔軟なモデル選択
- MCP 対応
- 多数の LLM Provider への接続
- LSP 統合
- セッション共有
- スクリプトとの統合

その上で、OpenCode と異なり、非エンジニアでも扱える GUI を追加し、Cowork のような UI/UX に近づける。

また、OpenClaw 系の運用思想も一部参考にする。特に次の点は Zero-Employee Orchestrator と相性がよい。

- 実行担当 AI と監督担当 AI の責務分離
- 改善提案や振り返りを append-only で蓄積する運用
- 自動で設定変更を反映しない慎重な改善フロー
- 秘密情報をメモリや改善ログに残さない原則
- 最小権限を前提とした運用設計

ただし、これはあくまで「自己改善ループ」や「監督 AI の運用原則」の参考であり、Zero-Employee Orchestrator 全体のアーキテクチャをそのまま流用するものではない。

### 35.4 UI / 使用感の要望

- GUI は Cowork 風、CLI / TUI は OpenCode 風にする。
- Skills / Plugins を一覧から追加・削除・確認できるようにする。
- チャットから業務用の Skills / Plugins を作成・修正できるようにする。
- 業務状況、進捗、コスト、使用 API をユーザーが常に確認できるようにする。
- AI が API を自動選定しても、ユーザーがコスト重視・品質重視・無料枠重視を選べるようにする。
- ローカル・クラウド・各種サービスに広く接続できるようにする。

### 35.5 セキュリティ・懸念点

- 最低限どの機能を本体に入れ、何を Plugin 化するかの基準を明確にする必要がある。
- Skill / Plugin をチャットで生成できても、安全性検証なしには導入させない。
- 大規模システムの置き換えでは、要件充足と保守運用の証明が必要になる。
- 同意画面フィッシングや危険な設定ファイルを含むリポジトリの取り扱いに注意する。
- Skill マージ前に静的解析や秘密情報漏洩チェックを入れる。
- OAuth は初期設定で必須にせず、必要に応じて Extension で有効化する方向も検討する。
- API キーやトークンの実値は、会話履歴・改善ログ・共有用 Skill / Plugin に残さない。
- 外部サービスの認証情報は、無条件に自動更新・自動反映しない。
- 認証情報の更新は、人間承認を前提とした安全なローテーション支援として設計する。

### 35.6 認証情報ローテーション方針

Zero-Employee Orchestrator は、外部 AI サービスやクラウド連携に用いる認証情報について、「自動更新」ではなく「安全なローテーション支援」を基本方針とする。

#### 基本原則

- API キー、OAuth、サービスアカウントなど、認証方式の違いを区別して管理する。
- 期限監視、権限棚卸し、更新提案、反映確認を一連のフローとして扱う。
- 料金、権限、公開設定、本番影響がある変更は必ず人間承認を必要とする。
- 秘密情報の実値は表示・保存・共有を最小化し、原則として Secret Manager や OS 標準の安全な保管先に置く。

#### 想定フロー

1. 接続先ごとに認証方式、権限範囲、作成日、最終確認日、適用環境を台帳化する。  
2. 一定期間ごとに、長期未更新、過剰権限、開発・本番共用などのリスクを検出する。  
3. ユーザーに対して、更新推奨理由、影響範囲、差し替え箇所、ロールバック手順を提示する。  
4. 人間承認後に、新しい認証情報の発行手順や差し替え手順を案内する。  
5. 更新後は疎通確認テストを行い、問題がなければ旧キーの無効化を促す。

#### 実装方針

- 外部 SaaS の API キーを一律に自動再発行する機能は、初期段階では持たない。
- 自動化する場合も、自社管理下の内部 API や Secret Manager 連携済みの資格情報など、限定領域から始める。
- Zero-Employee Orchestrator 本体では、認証情報の棚卸し、期限監視、更新提案、疎通確認、監査ログを中核機能とし、サービス別の詳細フローは Plugin / Extension で拡張する。

#### 想定 Skill / Plugin 例

- Credential Hygiene Plugin
- Secret Rotation Assistant Skill
- Post-Rotation Verification Skill

### 35.7 AI エージェント運用の要望

### 35.10 メタスキルの導入

現在の AI エージェントには、ハードスキルやソフトスキルはあっても、学び方・見通し方・自己修正のようなメタスキルが弱い。

### 35.13 AI 共創リパーパス（Co-creation Repurpose）

ユーザーが持つ1つの「素材」（ブログ記事、音声配信、ネタメモなど）から、AI 組織が複数メディア向けコンテンツを自動生成する構造を Plugin として実現したい。

- 音声 → テキスト → ブログ記事 → SNS 投稿 → 動画台本 → サムネイル案のように変換・展開する
- 「人間は縦に掘る。AI は横に広げる」を基本原則とする
- 一行タイトルからスライド自動生成、構成ルールに基づく講座コンテンツの組み立てなども含む
- 運用を繰り返すほど出力品質が向上し、自律的に分析・改善を行う
- 投稿・公開は必ず人間の最終チェックが必要

### 35.14 共進化ループ

AI 組織とユーザーの相互成長を設計思想として持つ。

- **User ↔ AI 共進化**: AI 秘書がユーザーの判断スタイルや価値観を学習するほど、ユーザー自身も AI を通じて知識や方法を獲得して育つ
- **AI ↔ AI 共進化**: AI 同士が互いの成果物や判断を参照し、組織としての精度が継続的に向上する
- **A2A（Agent-to-Agent）対話**: AI 組織間での情報共有、共同判断、業務委譲を可能にする
- 分身 AI はこの共進化ループの産物であり、単なるパラメータコピーではなく運用を通じて形成される

### 35.15 意思決定管理リポジトリ

プロジェクトオーナー（ユーザー）と AI 組織の意思決定を構造化して保存する場所を設ける。

- 「なぜこの機能が必要なのか」「なぜあえてこの機能は実装しなかったのか」をストーリーとして記録する
- ロードマップ構築、振り返り、方針転換時の参照基盤として活用する
- 会話ログとは別に、判断根拠・却下理由・代替案の記録を分離して保持する
- 将来的にはこの意思決定履歴を元に、AI が方針提案や優先順位提案を行えるようにする

### 35.16 ウェブブラウザ AI エージェントによるコスト最適化

ブラウザ操作型の AI エージェントを活用し、API 課金を回避する選択肢を提供する。

- ユーザーが保有する AI サービスのアカウント（無料枠含む）をブラウザ操作経由で使う
- 使用前に必ずアカウント利用の許可をユーザーに確認する
- 利用規約に違反しない範囲で運用し、公式 API が利用可能な場合はそちらを優先する
- ブラウザ操作は不安定性を伴うため、重要業務には API を推奨する旨を計画段階で明示する

### 35.17 セキュリティ・レッドチーミング

AI 組織内部に防御的セキュリティテスト機能を持たせる。

- AI 組織内のホワイトハッカー班が、自組織に対して模擬攻撃（情報漏洩しない偽のハッキングやマルウェア）を仕掛ける
- セキュリティ班がそれを検知・排除・対策する訓練を定期的に実施する
- この一連の防御テストを定期実行する Plugin として構築可能にする
- 外部や第三者を対象とする攻撃的用途は一切許可しない

### 35.19 AI の段階的制作アプローチ

- これはメタスキルの一環であり、「どう作るか」の方法論自体を AI が改善し続ける構造に寄与する

## 36. Paperclip 由来の中核要素を Zero-Employee Orchestrator に明文化して引き継ぐための追補

本節は、Zero-Employee Orchestrator が「AI を組織として運用する」思想をより実装可能な形で固定するための追補である。  
既存の思想や安全原則を維持したまま、Paperclip に見られる強みのうち、Zero-Employee Orchestrator と相性が良く、かつ本プロジェクトの方向性を壊さないものを明文化して取り込む。

### 36.1 Org Chart を中心オブジェクトとして扱う

Zero-Employee Orchestrator では、単に複数 AI を並列実行するのではなく、**会社・部署・チーム・役割・報告系統**を持った組織として扱う。  
そのため、Agent Team は以下の属性を持つ中心オブジェクトとする。

- 所属会社
- 所属部署
- 役職
- 上位責任者
- 委譲可能範囲
- 承認権限
- 利用可能 Skill / Plugin / Tool
- 予算上限
- 実行モード
- Heartbeat 設定
- 監査対象区分

これにより、誰が誰に報告し、どこまでを自律判断でき、どこから人間承認が必要かを、会話ではなく構造として保持する。

### 36.2 Goal Alignment を会社目標から task まで貫通させる

すべての案件、チケット、タスク、成果物は、可能な限り次の親子関係を持つ。

- Company Mission
- Strategic Goal
- Project
- Ticket / Work Item
- Task
- Artifact / Output

この系譜を持つことで、AI が「何をするか」だけでなく「なぜそれをするか」を理解した状態で計画・再計画・差し戻し対応を行えるようにする。

### 36.3 Heartbeat を補助機能ではなく運用原理として扱う

Zero-Employee Orchestrator では Heartbeat を単なる定期実行ではなく、**AI 組織が定期巡回して仕事の状態を確認し、必要に応じて進行・委譲・再計画・報告を行う運用原理**として扱う。  
Heartbeat は以下の契機で発火する。

- 定期スケジュール
- チケット新規作成
- タスク割当
- 差し戻し
- 外部イベント受信
- 予算残量閾値到達
- 承認待ち解除
- 上位者からの再指示
- 依存タスク完了

Heartbeat 実行時は、対象 Agent が次を行う。

- 自分の未完了タスク確認
- 依存関係の確認
- 予算・権限・期限の確認
- 必要であれば下位への委譲
- 必要であれば上位へのエスカレーション
- 必要であればユーザー承認要求
- 実行ログ・状態更新

### 36.4 Ticket / Work Item を日常運用の主単位にする

Zero-Employee Orchestrator では `spec / plan / tasks` を中間成果物として保持するが、日常運用の単位としては **Ticket / Work Item** を持つ。  
1つの依頼、案件、差し戻し、定期業務、障害対応、改善要求は、原則としてチケットとして起票される。

チケットは最低限次を持つ。

- 件名
- 起票者
- 所属会社 / プロジェクト
- 目的
- 優先度
- 状態
- 担当 Agent / Team / Human
- 関連する spec / plan / tasks
- 関連成果物
- 関連ログ
- 承認状態
- 期限
- 予算消費
- スレッド履歴

### 36.5 Multi-Company を初期設計から考慮する

Zero-Employee Orchestrator は、単一会社向けに始めても、初期スキーマの段階から **company_id** を主要エンティティへ持たせる。  
これにより、以下を成立させる。

- 1インスタンスで複数会社を運用
- 会社ごとの権限・ログ・予算・認証情報の分離
- テンプレート化した組織構造の複製
- 会社単位のバックアップ / エクスポート
- 将来的なマルチテナント化

### 36.6 Governance を「権限」だけでなく「取締役会モデル」として扱う

人間ユーザーは単なる管理者ではなく、**Board / Owner / Executive** として AI 組織を統治する。  
そのため、次は原則としてボード承認対象とする。

- 新規 Agent / Team の作成
- 権限昇格
- 予算上限変更
- 新しい外部連携の有効化
- 投稿 / 送信 / 公開 / 課金確定
- 認証情報変更
- Policy Pack の変更
- 自律実行範囲の拡張
- 削除系の危険操作
- 外部ファイル / DB / API への書き込み

### 36.7 ダッシュボードを運用の中心画面として扱う

Zero-Employee Orchestrator は、CLI / TUI / GUI を併存させるが、GUI においては次の情報を一画面で俯瞰できることを重要要件とする。

- 会社全体のミッションと現在の重点案件
- 進行中チケット一覧
- 承認待ち一覧
- Agent / Team の稼働状態
- Heartbeat 実行状況
- コスト消費状況
- エラー / 停止 / 差し戻し
- 直近成果物
- 推奨される次アクション

### 36.8 Bring Your Own Agent を前提にする

Zero-Employee Orchestrator は、特定の単一実行基盤に閉じない。  
Claude Code、Gemini、OpenAI API、ローカルモデル、スクリプト、HTTP Webhook、将来的なロボット・外部システムなど、**Heartbeat を受けて仕事を進められる単位**は接続可能対象とする。

| cancelled | 中止 | reopened |

## 44. テスト観点初期案

### 44.1 単体テスト
- repository の CRUD
- service の状態遷移
- policy 判定
- budget 計算
- heartbeat スケジュール解釈
- provider adapter の正常 / 異常系

### 44.2 結合テスト
- ticket 作成から task 実行までの一連フロー
- plan 承認後のみ実行可能であること
- approval reject 時に危険操作が走らないこと
- agent pause 時に新規実行されないこと
- budget 超過時に自動停止すること
- audit log が重要イベントを必ず記録すること

### 44.3 E2E テスト
- ログイン → 会社選択 → チケット作成 → plan 承認 → 実行 → 成果物確認
- 承認待ち画面で approve / reject
- 組織図画面で Agent の停止 / 再開
- ダッシュボードで異常状態が見えること
- Skill / Plugin 導入から実行までの導線

### 44.4 状態機械テスト
- Ticket 状態遷移の不正遷移拒否
- Task 状態遷移の不正遷移拒否
- Approval 状態遷移の多重実行防止
- Heartbeat 実行中の二重起動防止

### 44.5 セキュリティテスト
- company_id 越境アクセス拒否
- 権限のないユーザーによる承認拒否
- secret 値の平文露出防止
- 危険操作の承認バイパス不可
- tool input のサニタイズ
- Webhook 署名検証

### 44.6 可観測性テスト
- すべての重要イベントに trace_id が付与されること
- task_run と tool_call_trace が辿れること
- audit log から ticket / task / approval に到達できること
- コスト集計が日次 / 月次で一致すること

### 44.7 負荷・耐障害性テスト
- 1000 件の audit log 検索性能
- 同時 Heartbeat 実行時の競合
- 複数 Agent の同時 ticket checkout
- DB 再接続時のワーカー復旧
- Provider タイムアウト時の再試行と停止

### 44.8 LLM / Judge 特有のテスト
- 同一入力での plan 生成安定性評価
- 差し戻し時の再提案品質評価
- ポリシー違反検知の recall / precision
- Cross-Model Judge の一致率計測
- hallucination を含む成果物のレビュー捕捉率

### 44.9 受け入れテスト基準

初期MVPでは、最低でも次を満たした時点で「実用に向けた骨格が成立した」と判断する。

- 会社を作成できる
- Agent / Team を組織図で管理できる
- チケットを起票し、spec / plan / tasks を生成できる
- plan 承認後のみ task が走る
- 重要操作は approval を通る
- 実行結果と監査ログを追跡できる
- 予算超過時に実行停止できる
- Heartbeat で定期巡回できる

## 45. AI コーディングエージェントへの実装指示原則

この文書をもとに実装する AI エージェントは、以下の原則に従う。

1. 既存節を削除せず、本節以降の追補を優先参照して具体化すること  
2. まず DB モデル、状態遷移、API 契約を固定してから UI 実装に進むこと  
3. 重大操作では approval gate を先に実装し、後から緩めないこと  
4. 監査ログと trace_id を後回しにしないこと  
5. チケット、spec、plan、task、approval、artifact の関係が崩れる実装を避けること  
6. company_id による分離を全主要クエリで徹底すること  
7. stub / mock 実装でも API 契約と状態遷移は守ること  
8. YouTube など個別 Plugin を先に肥大化させず、基盤完成を優先すること  
9. CLI / TUI / GUI のどれから操作しても同じ状態が見える構造を維持すること  
10. 将来の Paperclip 的な org chart / heartbeat / budget / governance / multi-company を受け止められるよう、初期実装から骨格を壊さないこと
