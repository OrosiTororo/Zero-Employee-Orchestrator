# v0.1.6 Release Notes — MCP Integration & AI Team Roles

> Release date: 2026-04-10
> Previous release: [v0.1.5](v0.1.5.md) (2026-04-08)

## Headline

**Zero-Employee Orchestrator is now a fully spec-compliant Model Context
Protocol server.** Claude Desktop, Cursor, Continue, and every other
MCP-aware client can plug directly into ZEO and drive tickets, skills,
knowledge, audit, kill-switch, autonomy, budgets, and hypothesis
verification — without any custom glue code.

This release also introduces two permanent role definitions — the
**Skills Engineer** and the **Construction Engineer** — so that human
contributors and the ZEO self-improvement loop extend the platform in a
predictable, auditable way.

---

## New — Full Model Context Protocol (MCP) server

### JSON-RPC 2.0 transport

ZEO v0.1.5 shipped an MCP-flavored REST wrapper. v0.1.6 ships the real
wire protocol:

- **`POST /api/v1/mcp/rpc`** — JSON-RPC 2.0 endpoint per the
  [MCP 2024-11-05 specification](https://modelcontextprotocol.io/specification).
  Implements `initialize`, `ping`, `tools/list`, `tools/call`,
  `resources/list`, `resources/read`, `prompts/list`, `prompts/get`.
- **Batch requests** — the endpoint accepts JSON-RPC batches (list of
  requests) and returns one response per item.
- **Notifications** — notification-style calls (requests without an
  `id`) return HTTP 204 per the spec, so notification-heavy clients
  don't pollute logs with fake "responses".
- **Error codes** — `-32700` parse error, `-32600` invalid request,
  `-32601` method not found, `-32602` invalid params, `-32603` internal
  error, all emitted exactly as the JSON-RPC 2.0 spec prescribes.

### Streaming transport

- **`GET /api/v1/mcp/sse`** — Server-Sent Events endpoint. Emits a
  `ready` event containing the full capability manifest, then heartbeats
  every 15 s until the client disconnects. This is the path that
  subscribe-style clients like Cursor's MCP plugin use.

### 14 built-in MCP tools (up from 8)

| Tool | What it does |
|---|---|
| `create_ticket` | Create a new business ticket |
| `list_tickets` | Retrieve the current ticket list |
| `get_agent_status` | Read a running agent's live status |
| `search_knowledge` | Search the persistent knowledge base (Experience Memory) |
| `execute_skill` | Dispatch a registered skill by slug |
| `list_skills` | **(new)** List every registered skill, optionally enabled-only |
| `get_audit_logs` | Retrieve audit log tail with optional action-type filter |
| `monitor_executions` | Inspect in-flight task executions |
| `propose_hypothesis` | Propose + kick off parallel verification |
| `get_kill_switch_status` | **(new)** Global kill-switch state |
| `get_autonomy_level` | **(new)** Current autonomy dial level |
| `get_budget_status` | **(new)** Cost Guard budget usage |
| `list_approvals` | **(new)** Pending approval queue |
| `get_server_info` | **(new)** Version, tool count, protocol version |

### 6 resources + 3 prompts

Resources: `zero-employee://dashboard`, `/agents`, `/skills`,
`/knowledge`, `/kill-switch` *(new)*, `/autonomy` *(new)*.

Prompts: `task_planning`, `code_review`, `security_audit` *(new)*.

### Dynamic version resolution

The MCP `initialize` handshake used to report a hard-coded
`"version": "0.1.0"`, which drifted from every release. v0.1.6
resolves the version at runtime from `importlib.metadata` (or
`pyproject.toml` as a dev-checkout fallback) so the MCP protocol
always mirrors the pip-installed build.

### CLI integration

The CLI has a new top-level `mcp` subcommand for inspection and
smoke testing:

```bash
zero-employee mcp info              # Protocol version, counts, endpoints
zero-employee mcp tools             # Pretty-print every tool + description
zero-employee mcp call get_server_info
zero-employee mcp call create_ticket --args '{"title":"Test from CLI"}'
```

### Capability endpoint fix

Before v0.1.6, `GET /api/v1/mcp/capabilities` returned HTTP 500
because the Pydantic response model declared `tools: list[dict]`
while `MCPServer.get_capabilities()` returned `tools: list[str]`
(tool names). The response model has been corrected and the endpoint
now returns the full capability summary, including `protocol_version`
and `server_version`.

## New — AI Team Role Definitions

`skills-engineer.md` and `construction-engineer.md` at the repo root
have been rewritten from their original standalone Japanese essays
into **ZEO-specific English role definitions**. They are now the
canonical source of truth for how humans and the ZEO self-improvement
loop extend the platform.

### Skills Engineer (`skills-engineer.md`)

Owns **persona contracts** — the `SKILL.md` manifests that define the
tone, vocabulary, forbidden actions, and escalation policy of every
Skill ZEO calls.

Ships a canonical six-section `SKILL.md` template
(Persona, Inputs, Outputs, Tone & Style Rules, Forbidden Actions,
Escalation Policy), a pre-merge checklist, and a "one file per
responsibility" rule that keeps `CLAUDE.md` under 200 lines by
externalizing role details into the Skill registry.

### Construction Engineer (`construction-engineer.md`)

Owns **topology** — DAG graphs, judge pairs, approval placement,
autonomy thresholds, cost envelopes, kill-switch predicates, and
rollback procedures.

Ships a design-conversation template (7 stakeholder questions + a
one-page topology sketch YAML + a scoring rubric) and a drop-in
**"ZEO Construction Architect"** prompt that non-engineers can paste
into Claude Desktop, Cursor, or any MCP-aware client to design a
workflow *outside* the ZEO runtime before shipping it.

### Role separation

The two documents explicitly mirror each other: the Skills Engineer
never edits topology, the Construction Engineer never edits persona
text, and each handoff is reviewed by the counterpart. This ends the
historical drift where orchestration rules leaked into Skill personas
and vice versa.

## Upgrading

```bash
git pull origin main
pip install -U zero-employee-orchestrator   # once published to PyPI
# or
pip install -e apps/api                     # from source

zero-employee mcp info                      # verify v0.1.6 handshake
pytest apps/api/app/tests/test_mcp_server.py  # 16 MCP tests
```

For MCP-aware clients (Claude Desktop, Cursor, Continue), point them
at `http://localhost:18234/api/v1/mcp/rpc` once `zero-employee serve`
is running. No additional glue code required.
